Віртуальні потоки

Останнє оновлення: 16.10.2025

По умолчанию поток Java виконується в потоке платформы, предоставляемом операционной системой, в которой виконується виртуальная машина Java. Для многих рабочих нагрузок це приемлемое решение. Однак потоки платформы не є легковесными. На створення потока платформы выделяется множество ресурсов и времени. Це ограничивает количество потоков платформы, которые может обработать процессор. А приложениях с интенсивной нагрузкой це становится узким местом, ограничивая производительность приложения. Так, нередко программа выполняет такие операции, которые можуть занять продолжительное час, наприклад, обращение к сетевым ресурсам, чтение-запись файлов, обращение к базе данных и т.д. Такие операции можуть серьезно нагрузить приложение. Особенно це актуально в высоконагрвженных веб-приложениях, которые повинні быть готовы обслуживать тысячи запросов в секунду, и где продолжительные операции можуть блокировать інтерфейс пользователя и негативно повлиять на опыт пользователей. В случае с обычными потоками платформы при выполнении продолжительных операций поток просто бы блокировался на час виконання операции. Віртуальні потоки представляють решение цей проблемы.

Віртуальні потоки — це легковесные потоки, управляемые виртуальной машиной Java (JVM), а не операционной системой. Виртуальным потокам обычно требуется мало ресурсов, и одна виртуальная машина Java может поддерживать миллионы виртуальных потоков. Віртуальні потоки подходят для виконання задач, которые большую часть времени блокуються, наприклад, операции ввода-вывода - обращение к файлам, к базе данных, операции отправки сетевых запросов. Однак виртуальные потоки не призначенийы для длительных операций, интенсивно использующих процессор.

Как работает виртуальный поток? Виртуальный поток прикрепляется к потоку платформы и начинает выполнять неякий код. То есть для виконання виртуального потока все равно нвжен платформенный поток, який в данном случае ще нызвается потоком-носителем (carrier), оскільки условно "несет" в себе виртуальный поток. Когда код виртуального потока вызывает блокирующую операцию (наприклад, чтение файла), JVM не блокує несущий платформенный поток, к которому прикреплен виртуальный. Вместо цього среда виконання открепляет виртуальный поток с несущего потока платформы и сохраняет его состояние в куче, поки виконується блокирующая операция (наприклад, чтение файла). В це час несущий поток платформы звільняєся и может выполнять другие виртуальные потоки, другие действия. Как лише блокирующая операция завершуєся (наприклад, чтение файла завершено), JVM снова прикрепляет виртуальный поток обратно на доступный поток платформы для проповинения виконання. Цет механизм дозволяє эффективно утилизировать системные ресурсы, оскільки платформенные потоки не простаивают в ожидании, а постоянно выполняют полезную работу.

В общем случае це выглядит так. Наприклад, приходит запрос к веб-приложению, и для обработки запроса треба считать файл. Для обработки запроса запускаєся виртуальный поток, який прикрепляется к потоку платформы. Далїї цей поток видит, что треба считать файл, соответственно запускаєся операция чтения файла, которая может занять долгое час. Виртуальный поток ждет, поки буде считан файл, и открепляется от потока платформы, к которому он був прикреплен и в рамках которого он выполнялся. Однак поток платформы не ждет завершения чтения файла, а переключается на выполнение другой работы (при її наличие) - к нему может быть прикрпелен другой виртуальный поток. А когда файл буде считан, виртуальный поток возобновляет обработку и для цього снова прикрепляется к потоку платформы. Таким чином, приложение может болїї эффективно використовувати ресурсы процессора, представленные потоками платформы.

Основные преимущества виртуальных потоков

  • Масштабируемость: поскольку виртуальные потоки очень легковесны (занимают всего декілька сотен байт памяти по сравнению с мегабайтами для платформенных потоков), можна создавать миллионы таких потоков без существенной нагрузки на систему.

  • Простота кода: виртуальные потоки дозволяють писать код в привычном стиле, используя тот же API

  • Совместимость: виртуальные потоки є реализацией класса java.lang.Thread, что забезпечує их совместимость с существующим кодом и библиотеками, которые работают с потоками.

Створення виртуального потока

Віртуальні потоки имеют тот же API, что и потоки платформы. Для создания виртуальных потоков можна використовувати ряд способов:

  • С помощью статического метода Thread.startVirtualThread(Runnable task), в який передаєся выполняемая задача Runnable

  • С помощью статического метода Thread.Builder.OfVirtual.ofVirtual(), який створює строитель виртуального потока / фабрики потока

Рассмотрим оба цих способа.

Метод Thread.startVirtualThread

Самый простой способ создать и запустить виртуальный поток представляє статический метод Thread.startVirtualThread(), в який передаєся об'єкт Runnable:

public static Thread startVirtualThread(Runnable task)

Простейший пример:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {
            System.out.println("Hello from a virtual thread");
        };

        Thread.startVirtualThread(task); // запускаем виртуальный поток

        System.out.println("Main thread finished...");
    }
}

В данном случае в виртуальном потоке просто выводим сообщение на консоль. Однак вполне возможна при выполнении программы мы не увидим цього сообщения:


Main thread started...
Main thread finished...

По подобному консольному выводу не видно, чтобы виртуальный поток выполнялся. Все потому что, в отличие от платформенных потоков виртуальные потоки по умолчанию визначаються як потоки-демоны, которые выполняются в фоновом режиме. Однак поскольку виртуальные потоки используют тот же API, что и обычные, мы можемо вызвать метод join(), чтобы дождаться завершения виртуального потока:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {
            System.out.println("Hello from a virtual thread");
        };

        var t = Thread.startVirtualThread(task); // запускаем виртуальный поток
        try{
            t.join();       // ждем завершения виртуального потока
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

И теперь мы получим следующий консольный вывод:


Main thread started...
Hello from a virtual thread
Main thread finished...

Метод ofVirtual()

Метод ofVirtual() повертає строитель виртуального потока, який дозволяє выполнить болїї тонкую настройку:

public static Thread.Builder.OfVirtual ofVirtual()

Сам строитель представляє об'єкт типу Thread.Builder.OfVirtual, у которого мы можемо вызвать ряд методів:

  • Thread.Builder.OfVirtual name(String name)

    Устанавливает имя для потока

  • Thread.Builder.OfVirtual name(String prefix, long start)

    Також устанавливает имя для потока и плюс приймає значение счетчика, которое додаєся к имени потока

  • Thread start(Runnable task)

    Запускає виртуальный поток, возвращая его в виде об'єкта Thread

  • Thread unstarted(Runnable task)

    Створює новый поток, но без его запуска. Для планирования виконання потока у об'єкта Thread необхідно явным чином вызвать метод start().

  • ThreadFactory factory()

    Повертає фабрику создания потоков в виде об'єкта ThreadFactory

Используем цей метод для создания потока:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {
            System.out.println("Hello from " + Thread.currentThread().getName());
        };

        var t = Thread.ofVirtual()
                    .name("MyTask")  // устанавливаем имя
                    .start(task);     // запускаем поток
        try{
            t.join();       // ждем завершения виртуального потока
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

Консольный вывод:


Main thread started...
Hello from MyTask
Main thread finished...

Или можна сначала создать, а потом отдельно запустить:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        var t = Thread.ofVirtual()
                    .name("MyTask")  // устанавливаем имя
                    .unstarted(() -> System.out.println("Hello from " + Thread.currentThread().getName()));

        t.start();     // запускаем поток
        try{
            t.join();       // ждем завершения виртуального потока
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

Особенности свойств виртуального потока

Для виртуальных потоков метод isVirtual() повертає true

Варто відзначити, что по умолчанию у виртуальных потоков нет имени потока. Метод getName() повертає пустую строку, якщо имя потока не задано. Тому якщо необхідно разграничить декілька потоков на основе имени, йде для них устанавливать имя явным чином.

Віртуальні потоки є потоками-демонами и, следовательно, не препятствуют началу последовательности завершения работы. Тому для них метод isDaemon() повертає true

Віртуальні потоки имеют фиксированный приоритет потока, який нельзя изменить. Посмотрим на эти свойства потока:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {

            Thread t = Thread.currentThread(); // получаем текущий поток

            System.out.println("Name: " + t.getName()); // якщо имя не задано, то t.getName() вернет пустую строку
            System.out.println("Is Vitual: " + t.isVirtual());
            System.out.println("Is Daemon: " + t.isDaemon());
            System.out.println("Priority: " + t.getPriority());
        };

        var thread = Thread.startVirtualThread(task);

        try{
            thread.join();       // ждем завершения виртуального потока
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

Консольный вывод:


Main thread started...
Name: 
Is Vitual: true
Is Daemon: true
Priority 5
Main thread finished...

Когда варто и не варто використовувати виртуальные потоки?

Идеальные кандидаты для использования виртуальных потоков:

  • I/O-bound задачи (работа с подсистемой ввода-вывода): приложения с большим количеством блокирующих операций ввода-вывода, такие як веб-серверы, микросервисы, приложения для работы с базами данных

  • Высоконагрвженные системы: системы, которым необхідно обробляти тысячи або миллионы одновременных подключений

Когда лучше воздержаться от виртуальных потоков:

  • CPU-bound задачи: для задач, интенсивно использующих процессор (наприклад, сложные математические вычисления, обработка больших обсягов данных в памяти), преимущества виртуальных потоков не так заметны. В цем случае количество потоков повинно быть соизмеримо с количеством ядер процессора, и классические платформенные потоки подойдут лучше

  • Работа с synchronized-блоками: длительное удержание блокировок в synchronized-блоках (рассматриваются в следующих статьях) может "прибить" (pin) виртуальный поток к его несущему платформенному потоку, что не позволит последнему выполнять другие задачи и может привести к снижению производительности, поцему йде с осторожностью використовувати синхронизацию с synchronized

  • Вызовы нативных методів (JNI): аналогично synchronized, вызовы нативного кода можуть блокировать несущий поток
