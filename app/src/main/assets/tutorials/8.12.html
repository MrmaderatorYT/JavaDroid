Віртуальні потіки

Останнє оновлення: 16.10.2025

По умолчанию потік Java виконується у потоці платформы, предоставляемом операционной системой, в которой виконується виртуальная машина Java. Для многих рабочих нагрузок це приемлемое решение. Однак потіки платформы не є легковесными. На створення потіка платформы выделяется множина ресурсов и времени. Це ограничивает кількість потіков платформы, которые может обпрацювати процесор. А додатких с интенсивной нагрузкой це становится узким местом, ограничивая производительность додатки. Так, нередко програма виконує такие операції, которые можуть занять продолжительное час, наприклад, обращение к сетевым ресурсам, читання-запис файлів, обращение к базе данных и т.д. Такие операції можуть серьезно нагрузить додаток. Особенно це актуально в высоконагрвженных веб-додатких, которые повинні быть готовы обслуживать тысячи запросов в секунду, и где продолжительные операції можуть блокировать інтерфейс пользователя и негативно повлиять на опыт пользователей. В случае с обычными потіками платформы при выполнении продолжительных операцій потік просто бы блокировался на час виконання операції. Віртуальні потіки представляютьь решение цей проблемы.

Віртуальні потіки — це легковесные потіки, управляемые виртуальной машиной Java (JVM), а не операционной системой. Виртуальным потікам обычно требуется мало ресурсов, и одна виртуальная машина Java может піддерживать миллионы виртуальных потіков. Віртуальні потіки підходят для виконання завдань, которые большую часть времени блокуються, наприклад, операції ввода-вивіда - обращение к файлум, к базе данных, операції отправки сетевых запросов. Однак виртуальные потіки не призначенийы для длительных операцій, интенсивно использующих процесор.

Как работает виртуальный потік? Виртуальный потік прикрепляется к потіку платформы и начинает выполнять неякий код. То есть для виконання виртуального потіка все равно нвжен платформенный потік, який в данном случае ще нызвается потіком-носителем (carrier), оскільки условно "несет" в себе виртуальный потік. Когда код виртуального потіка вызывает блокирующую операцію (наприклад, читання файлу), JVM не блокує несущий платформенный потік, к которому прикреплен виртуальный. Вместо цього среда виконання открепляет виртуальный потік с несущего потіка платформы и сохраняет его состояние в куче, поки виконується блокирующая операція (наприклад, читання файлу). В це час несущий потік платформы звільняєся и может выполнять другие виртуальные потіки, другие действия. Как лише блокирующая операція завершуєся (наприклад, читання файлу завершено), JVM снова прикрепляет виртуальный потік обратно на доступный потік платформы для проповинения виконання. Цет механизм дозволяє эффективно утилизировать системные ресурсы, оскільки платформенные потіки не простаивают в ожидании, а постоянно виконують полезную работу.

В общем случае це выглядит так. Наприклад, приходит запрос к веб-приложению, и для обработки запроса треба считать файл. Для обработки запроса запускуєся виртуальный потік, який прикрепляется к потіку платформы. Далїї цей потік видит, что треба считать файл, соответственно запускуєся операція читання файлу, которая может занять долгое час. Виртуальный потік ждет, поки буде считан файл, и открепляется от потіка платформы, к которому он був прикреплен и в рамках которого он выполнялся. Однак потік платформы не ждет завершення читання файлу, а переключается на виконання другой работы (при її наличие) - к нему может быть прикрпелен другой виртуальный потік. А когда файл буде считан, виртуальный потік возобновляет обработку и для цього снова прикрепляется к потіку платформы. Таким чином, додаток может болїї эффективно використовувати ресурсы процесора, представленные потіками платформы.

Основные преимущества виртуальных потіков

  • Масштабируемость: поскольку виртуальные потіки дуже легковесны (занимают всего декілька сотен байт памяти по сравнению с мегабайтами для платформенных потіков), можна создавать миллионы таких потіков без существенной нагрузки на систему.

  • Простота коду: виртуальные потіки дозволяють писать код в привычном стиле, використовуючи тот же API

  • Совместимость: виртуальные потіки є реализацией класа java.lang.Thread, что забезпечує их совместимость с существующим кодом и бібліотеками, которые работают с потіками.

Створення виртуального потіка

Віртуальні потіки имеют тот же API, что и потіки платформы. Для створенняя виртуальных потіков можна використовувати ряд способов:

  • За допомогою статического методу Thread.startVirtualThread(Runnable task), в який передаєся выполняемая завданьа Runnable

  • За допомогою статического методу Thread.Builder.OfVirtual.ofVirtual(), який створює строитель виртуального потіка / фабрики потіка

Рассмотрим оба цих способа.

Метод Thread.startVirtualThread

Самый простой способ створити и запустить виртуальный потік представляє статический метод Thread.startVirtualThread(), в який передаєся об'єкт Runnable:

public static Thread startVirtualThread(Runnable task)

Простейший пример:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {
            System.out.println("Hello from a virtual thread");
        };

        Thread.startVirtualThread(task); // запускуємо виртуальный потік

        System.out.println("Main thread finished...");
    }
}

У даному випадку в виртуальном потіке просто вивідим сообщение на консоль. Однак вполне возможна при выполнении програми мы не увидим цього сообщения:


Main thread started...
Main thread finished...

По підобному консольному вивіду не видно, чтобы виртуальный потік выполнялся. Все потому что, в отличие от платформенных потіков виртуальные потіки по умолчанию визначаються як потіки-демоны, которые виконуються в фоновом режиме. Однак поскольку виртуальные потіки використовують тот же API, что и обычные, мы можемо вызвать метод join(), чтобы дождаться завершення виртуального потіка:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {
            System.out.println("Hello from a virtual thread");
        };

        var t = Thread.startVirtualThread(task); // запускуємо виртуальный потік
        try{
            t.join();       // ждем завершення виртуального потіка
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

И теперь мы получим следующий консольный вивід:


Main thread started...
Hello from a virtual thread
Main thread finished...

Метод ofVirtual()

Метод ofVirtual() повертає строитель виртуального потіка, який дозволяє выполнить болїї тонкую настройку:

public static Thread.Builder.OfVirtual ofVirtual()

Сам строитель представляє об'єкт типу Thread.Builder.OfVirtual, у которого мы можемо вызвать ряд методів:

  • Thread.Builder.OfVirtual name(String name)

    Встановлює ім'я для потіка

  • Thread.Builder.OfVirtual name(String prefix, long start)

    Також встановлює ім'я для потіка и плюс приймає значення счетчика, которое додаєся к імені потіка

  • Thread start(Runnable task)

    Запускує виртуальный потік, возвращая его в виде об'єкта Thread

  • Thread unstarted(Runnable task)

    Створює новый потік, но без его запуску. Для планирования виконання потіка у об'єкта Thread необхідно явным чином вызвать метод start().

  • ThreadFactory factory()

    Повертає фабрику створенняя потіков в виде об'єкта ThreadFactory

Використовуємо цей метод для створенняя потіка:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {
            System.out.println("Hello from " + Thread.currentThread().getName());
        };

        var t = Thread.ofVirtual()
                    .name("MyTask")  // устанавливаем ім'я
                    .start(task);     // запускуємо потік
        try{
            t.join();       // ждем завершення виртуального потіка
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

Консольный вивід:


Main thread started...
Hello from MyTask
Main thread finished...

Или можна сначала створити, а потом отдельно запустить:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        var t = Thread.ofVirtual()
                    .name("MyTask")  // устанавливаем ім'я
                    .unstarted(() -> System.out.println("Hello from " + Thread.currentThread().getName()));

        t.start();     // запускуємо потік
        try{
            t.join();       // ждем завершення виртуального потіка
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

Особенности свойств виртуального потіка

Для виртуальных потіков метод isVirtual() повертає true

Варто відзначити, что по умолчанию у виртуальных потіков нет імені потіка. Метод getName() повертає пустую рядок, якщо ім'я потіка не задано. Тому якщо необхідно разграничить декілька потіков на основе імені, йде для них устанавливать ім'я явным чином.

Віртуальні потіки є потіками-демонами и, следовательно, не препятствуют началу післядовательности завершення работы. Тому для них метод isDaemon() повертає true

Віртуальні потіки имеют фиксированный приоритет потіка, який нельзя змінити. Посмотрим на эти свойства потіка:


class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        Runnable task = () -> {

            Thread t = Thread.currentThread(); // отримуємо текущий потік

            System.out.println("Name: " + t.getName()); // якщо ім'я не задано, то t.getName() вернет пустую рядок
            System.out.println("Is Vitual: " + t.isVirtual());
            System.out.println("Is Daemon: " + t.isDaemon());
            System.out.println("Priority: " + t.getPriority());
        };

        var thread = Thread.startVirtualThread(task);

        try{
            thread.join();       // ждем завершення виртуального потіка
        }
        catch(InterruptedException _){
            System.out.println("Main thread interrupted");
        }

        System.out.println("Main thread finished...");
    }
}

Консольный вивід:


Main thread started...
Name: 
Is Vitual: true
Is Daemon: true
Priority 5
Main thread finished...

Когда варто и не варто використовувати виртуальные потіки?

Идеальные кандидаты для использования виртуальных потіков:

  • I/O-bound завданьі (работа с підсистемой ввода-вивіда): додатки с большим кількістьм блокирующих операцій ввода-вивіда, такие як веб-серверы, микросервисы, додатки для работы с базами данных

  • Высоконагрвженные системы: системы, которым необхідно обробляти тысячи або миллионы одновременных підключений

Когда лучше воздержаться от виртуальных потіков:

  • CPU-bound завданьі: для завдань, интенсивно использующих процесор (наприклад, сложные математические вычисления, обробка больших обсягов данных в памяти), преимущества виртуальных потіков не так заметны. В цем случае кількість потіков повинно быть соизмеримо с кількістьм ядер процесора, и класические платформенные потіки підойдут лучше

  • Работа с synchronized-блоками: длительное удержание блокировок в synchronized-блоках (рассматриваются в следующих статьях) может "прибить" (pin) виртуальный потік к его несущему платформенному потіку, что не дозволить післяднему выполнять другие завданьі и может привести к снижению производительности, поцему йде с осторожностью використовувати синхронизацию с synchronized

  • Вызовы нативных методів (JNI): аналогично synchronized, вызовы нативного коду можуть блокировать несущий потік
