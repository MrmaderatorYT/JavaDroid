Executor – виконавець завдань

Останнє оновлення: 20.10.2025

Executor або исполнитель представляє об'єкт, який виконує полученные завданьі. Исполнитель обычно використовується вместо явного створенняя потіков и дозволяє отделить отправку завданьі от механизма її виконання.

Executor представляє функциональный інтерфейс с одним методом execute():


void execute(Runnable command)

Предполагается, что данный метод приймає завданьу Runnable и виконує її в неякий момент в будущем. Простейший пример:


import java.util.concurrent.*;

class TaskExecutor implements Executor{
    
    public void execute(Runnable task){
        task.run();
    }
}

class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        var executor = new TaskExecutor();
        executor.execute(() -> {
                    
            System.out.println("Executing task in " + Thread.currentThread());
        });
        System.out.println("Main thread finished...");
    }
}

У даному випадку определяем клас TaskExecutor, який реалізує інтерфейс Executor и в котором метод execute() просто запускує переданную завданьу на виконання.

В методі main створюємо об'єкт цього класа и вызываем его метод execute(), передавая в него простешую завданьу Runnable. Таким чином виконується переданная завданьа.

В таком запуску завданьі, конечно, нет большого смысла, оскільки мы просто могли бы запустить ту же завданьу Runnable в том же методі main() без реалізації інтерфейса Executor. Но тем не менїї інтерфейс Executor не требует строго асинхронного виконання и может выполняться синхронно, як в примере выше, где завданьа выполнялась непосредственно у потоці методу main()

Однак чаще всего завданьі виконуються у потоці, отличном от потіка вызывающего об'єкта. Наприклад, изменим код исполнителя, чтобы он создавал новый потік для каждой завданьі:


class TaskExecutor implements Executor{
    
    public void execute(Runnable task){
        new Thread(task).start();
    }
}

При запуску програми мы увидим, что теперь завданьа виконується в отдельном потіке:


Main thread started...
Main thread finished...
Executing task in Thread[#34,Thread-0,5,main]

Нередко реалізації Executor використовуються для координации виконання завдань. И в большинстве случаев нам не потребуется создавать свои собственные реалізації інтерфейса Executor, оскільки в Java есть встроенные реалізації цього інтерфейса, наприклад, класа ThreadPoolExecutor, який для управления виконанням завдань використовує пул потіков, позволяя оптимизировать виконання програми.

ExecutorService

Інтерфейс ExecutorService расширяет інтерфейс Executor, предоставляя методи управления завершенням работы, а також методи створенняя об'єкта Future для отслеживания хода виконання одной або нескольких асинхронных завдань. Цет інтерфейс упрощает застосування исполнителей Executor для управления асинхронными завданьами. Выше упомянутый клас ThreadPoolExecutor як раз реалізує данный інтерфейс ExecutorService (а через него и Executor). Рассмотрим, як використовувати ExecutorService.

Створення исполнителя

Для створенняя исполнителей застосовується один из методів класа Executors. Підобные методи в основном создают пулы потіков. Пул потіков представляє технику для оптимизации работы с платформенными потіками. Поскольку створення платформенных потіков досить затратно и требует больших ресурсов, то вместо створенняя новых потіков болїї производительным решением є використання вже готовых потіков. И пул потіков представляє набор свободных потіков, готовых к выполнению завдань. Так, якщо треба выполнить у потоці якую-нибудь завданьу Runnable або Callable, то програма берет из пула один из потіков и використовує его для виконання завданьі. Після виконання завданьі потік не завершуєся, а повертаєся обратно в пул для обслуживания следующего запроса.

Некоторые из них:

  • Executors.newCachedThreadPool()

    Створює пул потіков, який створює новые потіки по мере необхідности, и повторно використовує ранїї создані потіки, когда они становятся доступны. Бездействующие потіки сохраняются в течение 60 секунд

  • Executors.newFixedThreadPool()

    Створює пул из фиксированного количества потіков; бездействующие потіки сохраняются неограниченное час. Если число отправленных завдань превышает число бездействующих потіков, необработанные завданьі помещаются в очередь. Они запускуються після завершення других завдань.

  • Executors.newSingleThreadExecutor()

    Створює об'єкт Executor (фактически пул из одного потіка) с одним рабочим потіком, який післядовательно виконує отправленные завданьі.

  • Executors.newVirtualThreadPerTaskExecutor()

    Створює об'єкт Executor, який запускує каждую завданьу в новом виртуальном потіке.

Все эти методи повертають об'єкты класов, которые реалізують інтерфейс ExecutorService, який в свою очередь расширяет інтерфейс Executor.

Надсилання завдань виконавцю

Для отправки завданьі Runnable або Callable в ExecutorService застосовується один из следующих методів:


Future<T> submit(Callable<T> task)
Future<?> submit(Runnable task)
Future<T> submit(Runnable task, T result)

При вызове методу submit() исполнитель - об'єкт Executor планирует виконання отправленной завданьі.

В качестве результата методи повертають об'єкт Future, який можна використовувати для получения результата або отмены завданьі.

Вторая версия методу submit повертає об'єкт Future<?>. Его метод get() блокує виконання до завершення завданьі, а затем повертає значення null.

Третья версия методу submit повертає об'єкт Future, у которого метод get() після завершення завданьі повертає об'єкт, переданный через второй параметр.

Завершення исполнителя

Після завершення работы исполнителя для освобождения ресурсов вызывается метод close():

void close()

Закрытый исполнитель не приймає новых завдань. Метод close() блокує виконання до завершення всех отправленных завдань. (Если потік, вызывающий close, переривається, отправленные завданьі отменяются.) Для болїї удобного неявного вызова методу close() можна використовувати конструкцию try-with-resources (try с ресурсами):


try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {

    ...............

    Future<V> f = submit(myCallable);

    .......................
} // здесь вызывается метод executor.close()

В итоге общий план применения об'єкта Executor для виконання завданьі:

  1. За допомогою одного из статических методів класа Executors (newCachedThreadPool, newFixedThreadPool и т.д) створюєся об'єкт ExectorService

  2. У об'єкта ExectorService для отправки завдань Callable або Runnable вызывается метод submit()

  3. Метод submit() повертає об'єкт Future. Цет об'єкт можна сохранить повертаємоые, чтобы иметь возможнасть запрашивать состояния завдань, получать результат завданьі або отменять її.

  4. В конце, якщо больше не планируется отправлять завданьі на виконання, у об'єкта ExectorService вызывается метод close(), який завершує работу исполнителя.

Рассмотрим простейший пример:


import java.util.concurrent.*;

class Program{

    public static void main(String[] args) {
         
        System.out.println("Main thread started...");

        try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {

            // запускуємо 5 завдань в виртуальных потіках
            for (int i = 0; i < 5; i++) {

                int taskNumber = i;

                executor.submit(() -> {
                    try{
                        Thread.sleep(1000);
                    }
                    catch(Exception ex) { 
                        System.out.println(ex.getMessage());
                    }
                    System.out.println("Executing task " + taskNumber + " in " + Thread.currentThread());
                });
            }
        } // executor.close() вызывается автоматически

        System.out.println("Main thread finished...");
    }
}

Здесь ExecutorService створює новый виртуальный потік для каждой завданьі Runnable:

try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {

Через метод submit() передаємо на виконання завданьу в виде лямбда-выражения:

executor.submit(() -> {
