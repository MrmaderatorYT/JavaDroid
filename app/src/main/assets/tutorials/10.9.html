Паралельні потіки

Останнє оновлення: 12.01.2022

Кроме післядовательных потіков Stream API піддерживает параллельные потіки. Распараллеливание потіков дозволяє задействовать декілька ядер процесора (якщо целевая машина многоядерная) и тем самым может повысить производительность и ускорить вычисления. В то же час говорить, что застосування параллельных потіков на многоядерных машинах однозначно повысит производительность - не совсем корректно. В каждом конкретном случае треба проверять и тестировать.

Щоб сделать обычный післядовательный потік параллельным, треба вызвать у об'єкта Stream метод parallel. Крім того, можна також використовувати метод parallelStream() інтерфейса Collection для створенняя параллельного потіка из колекції.

В то же час якщо рабочая машина не є многоядерной, то потік буде выполняться як післядовательный.

Застосування параллельных потіков во многих случаях буде аналогично. Наприклад:


import java.util.Optional;
import java.util.stream.Stream;

public class Program {
 
    public static void main(String[] args) {
		
		Stream<Integer> numbersStream = Stream.of(1, 2, 3, 4, 5, 6);
		Optional<Integer> result = numbersStream.parallel().reduce((x,y)-> x*y);
		System.out.println(result.get()); // 720
	}
}

Ще один пример:


import java.util.Arrays;
import java.util.List;

public class Program {

    public static void main(String[] args) {

        List<String> people = Arrays.asList("Tom","Bob", "Sam", "Kate", "Tim");

        System.out.println("Післядовательный потік");
        people.stream().filter(p->p.length()==3).forEach(System.out::println);

        System.out.println("\nПараллельный потік");
        people.parallelStream().filter(p->p.length()==3).forEach(System.out::println);
    }
}

У даному випадку сначала для списку people створюємо потік и выполняем над ним ряд операцій в післядовательном режиме. Зокрема, находим в списке рядки, длина которых равна 3 и вивідим их на консоль. В цем случае все операції с потіком будуть производиться над елементами списку в том порядке, в котоом елементы идут в списке.

Затем за допомогою методу people.parallelStream() для списку створюєся параллельный потік. Причем застосовуються те же операції, однак теперь порядок, в котором над елементами списку будуть производиться операції, не детерминирован.

Консольный вивід:

Післядовательный потік
Tom
Bob
Sam
Tim

Параллельный потік
Sam
Tim
Bob
Tom

В случае с параллельным потіком вивід недетерминирован и может отличаться.

Однак не все функції можна без ущерба для точности вычисления перенести с післядовательных потіков на параллельные. Прежде всего такие функції повинні быть без сохранения состояния и ассоциативными, то есть при выполнении слева направо давать тот же результат, что и при выполнении справа налево, як в случае с произведением чисел. Наприклад:


Stream<Integer> numbersStream = Stream.of(1, 2, 3, 4, 5, 6);
Integer result = numbersStream.parallel().reduce(1, (x,y)->x * y);
System.out.println(result);

Фактически здесь происходит перемножение чисел. При цем нет разницы між 1 * 2 * 3 * 4 * (5 * 6) або 5 * 6 * 1 * (2 * 3) * 4. Мы можемо расставить скобки любым чином, разместить післядовательность чисел в любом порядке, и все равно мы получим один и тот же результат. То есть данная операція є ассоциативной и поцему может быть распараллелена.

Підпросы производительности в параллельных операціях

Фактически застосування параллельных потіков сводится к тому, что дані у потоці будуть розділены на части, каждая часть обробляєся на отдельном ядре процесора, и в конце эти части соединяются, и над ними виконуються финальные операції. Рассмотрим некоторые критерии, которые можуть повлиять на производительность в параллельных потіках:

  • Размер данных. Чем больше данных, тем сложнїї сначала розділять дані, а потом их соединять.

  • Количество ядер процесора. Теоретически, чем больше ядер в компьютере, тем быстрїї програма буде працювати. Если на машине одно ядро, нет смысла застосовувати параллельные потіки.

  • Чем проще структура данных, с которой работает потік, тем быстрїї будуть происходить операції. Наприклад, дані из ArrayList легко використовувати, оскільки структура данной колекції предполагает післядовательность несвязанных данных. А вот колекція типу LinkedList - не лучший вариант, оскільки в післядовательном списке всі елементи связаны с предыдущими/понаступними. И такие дані трудно распараллелить.

  • Над данными примитивных типів операції будуть производиться быстрїї, чем над об'єктами класов

Упорядоченность в параллельных потіках

Как правило, елементы передаються в потік в том же порядке, в котором они визначенийы в источнике данных. При работе с параллельными потіками система сохраняет порядок следования елементов. Виняток составляет метод forEach(), який может виводитьь елементы в произвольном порядке. И чтобы сохранить порядок следования, необхідно застосовувати метод forEachOrdered:


phones.parallelStream()
    .sorted()
    .forEachOrdered(s->System.out.println(s));

Сохранение порядка в параллельных потіках увеличивает издержки при выполнении. Но якщо нам порядок не важен, то мы можемо отключить его сохранение и тем самым увеличить производительность, использовав метод unordered:


phones.parallelStream()
    .sorted()
	.unordered()
    .forEach(s->System.out.println(s));
