Багатопоточне програмування та клас Thread

Останнє оновлення: 14.10.2025

Большинство языков програмирования піддерживают такую важную функциональность як многопоточность, и Java в цем плане не виняток. При помощи многопоточности мы можемо выделить в додатку декілька потіков, которые будуть выполнять различные завданьі одновременно. Если у нас, допустим, графическое додаток, которое посылает запрос к якому-нибудь серверу або считывает и обробляє огромный файл, то без многопоточности у нас бы блокировался графический інтерфейс на час виконання завданьі. А благодаря потікам мы можемо выделить отправку запроса або любую другую завданьу, которая может долго оброблятися, в отдельный потік. Тому большинство реальных приложений, которые многим из нас приходится використовувати, практически не мыслимы без многопоточности.

Варто відзначити, что начиная с версии 21 в мові Java есть два типу потіков: потіки платформы (platform thread) и виртуальные потіки (virtual thread). Потік платформы представляє потік, предоставляемый операционной системой, которая планирует его виконання. Віртуальні потіки виконуються в потіках платформы и планируются средой виконання Java.

Каждый процесор может одновременно выполнять один потік платформы. На машине с несколькими процесорами можна запускуть декілька потіков параллельно. Если кількість потіков платформы больше, чем кількість доступных процесоров, то каждому запущенному потіку выделяется визначенийный промежуток времени для виконання своей завданьі. Когда цей промежуток времени исчерпывается, операционная система вытесняет потік и надає возможнасть іншому потіку выполнить его.

Клас Thread

В мові Java за работу с потіками отвечает клас Thread. Фактически цей клас и представляє функциональность потіка. И чтобы створити новый потік, нам треба створити об'єкт цього класа.

Но все потіки необхідно создавать вручую. Когда запускуєся програма, начинает працювати главный потік цей програми, в котором и виконується метод main(). От цього главного потіка порождаются все остальные дочерние потіки.

Отримання текущего потіка

За допомогою статического методу Thread.currentThread() мы можемо отримати текущий потік виконання в виде об'єкта Thread:


public class Program {
  
    public static void main(String[] args) {
          
        // отримуємо текущий потік
        Thread t = Thread.currentThread(); 
        System.out.println(t); // Thread[#3,main,5,main]
    }
}

У даному випадку мы отримуємо текущий потік, в котором запущен метод main, в змінну t и вивідим его на консоль. В итоге мы получим консольный вивід следующего вида:

Thread[#3,main,5,main]

Что значит цей вивід? Цет вивід сформирован за допомогою методу toString() и означает следующїї:

идентификатор_потіка, ім'я_потіка, приоритет, группа потіка

То есть мы отримуємо:

  • идентификатор потіка: 3

  • ім'я потіка: main

  • приоритет потіка: 5

  • группа потіка: main

Варто відзначити, что за допомогою статического методу Thread.getAllStackTraces() мы можемо отримати трассировку стека для всех активных потіков в програме в виде словника Map<Thread, StackTraceElement[]>. Наприклад, выведем на консоль все активные потіки:


public class Program {
  
    public static void main(String[] args) {
          
        // отримуємо лише потіки без трасировки стека
        var threads = Thread.getAllStackTraces().keySet(); 

        for(var thread : threads){
            System.out.println(thread); 
        }
    }
}

В цем случае мы получим консольный вивід что-то вроде следующего:


Thread[#3,main,5,main]
Thread[#15,Reference Handler,10,system]
Thread[#16,Finalizer,8,system]
Thread[#32,Notification Thread,9,system]
Thread[#17,Signal Dispatcher,9,system]
Thread[#33,Common-Cleaner,8,InnocuousThreadGroup]

Отримання информации о потіках

Клас Thread надає ряд методів для управления и получения состояния потіков. Отмечу некоторые из цих методів:

  • String getName(): повертає ім'я потіка

  • void setName(String name): встановлює ім'я потіка

  • int getPriority(): повертає приоритет потіка

  • void setPriority(int proirity): встановлює приоритет потіка. Приоритет є одним из ключевых факторов для выбора системой потіка из кучи потіков для виконання. В цей метод в качестве параметра передаєся числовое значення приоритета - от 1 до 10. По умолчанию главному потіку выставляется средний приоритет - 5.

  • StackTraceElement[] getStackTrace(): повертає трассировку стека текущего потіка в виде масива об'єктов StackTraceElement

  • void setDaemon(boolean on): встановлює, є ли потік фоновым

  • boolean isAlive(): повертає true, якщо потік активен

  • boolean isInterrupted(): повертає true, якщо потік був прерван

  • boolean isDaemon(): повертає true, якщо потік є фоновым (потіком-демоном)

  • boolean isVirtual(): повертає true, якщо потік є виртуальным

  • long threadId(): повертає идентификатор текущего потіка

И також можна выделить ряд методів по управлению потіками:

  • void join(): очікує завершення потіка

  • void run(): визначає действия, которые собственно и виконує потік

  • static void sleep(long millis): приостанавливает потік на заданное кількість миллисекунд

  • start(): запускує (или ставит в очередь на запуск) потік, вызывая его метод run()

  • static Thread startVirtualThread(Runnable task): запускує або ставит в очередь на запуск виртуальный потік

Наприклад, получим информацию о текущем потіке (главном потіке програми):


public class Program {
  
    public static void main(String[] args) {
          
        // отримуємо текущий потік
        Thread t = Thread.currentThread(); 
        // отримуємо его свойства

        System.out.println("Id: " + t.threadId()); 
        System.out.println("Name: " + t.getName()); 
        System.out.println("Priority: " + t.getPriority()); 
        System.out.println("State: " + t.getState()); 
        System.out.println("isAlive: " + t.isAlive()); 
        System.out.println("isDaemon: " + t.isDaemon()); 
        System.out.println("isVirtual: " + t.isVirtual()); 
    }
}

У даному випадку мы получим следующий консольный вивід:


Id: 3
Name: main
Priority: 5
State: RUNNABLE
isAlive: true
isDaemon: false
isVirtual: false

Рассмотрим некоторые свойства потіков підробнїї.

Состояние потіков

Потіки можуть находиться в следующих 6 состояниях:

  • NEW: потік создан, но ще не запущен

  • RUNNABLE: потік виконується

  • BLOCKED: потік заблокирован в ожидании блокировки монитора

  • WAITING: потік невизначенийно долго очікує, поки другой потік выполнит визначенийное действие

  • TIMED_WAITING: потік, ожидающий виконання действия другим потіком в течение заданного времени

  • TERMINATED: потік завершил виконання

В любой момент времени потік может находиться лише в одном состоянии. Эти состояния представляютьь собой состояния виртуальной машины, которые не отражают состояния потіков операционной системы.

Приоритеты потіков

Каждый потік имїїт приоритет. Приоритеты представляютьь числа от 1 до 10. Для их описания Java представляє три константи:

  • MIN_PRIORITY: минимальный приоритет, відповідає числу 1

  • MAX_PRIORITY: максимальный приоритет, відповідає числу 10

  • NORM_PRIORITY: стандартный приоритет, відповідає числу 5

По умолчанию потік найде приоритет потіка, який его создал. Но теоретически можна увеличить або уменьшить приоритет любого потіка платформы за допомогою методу setPriority(). Зокрема, можна установить приоритет на любое значення між MIN_PRIORITY (1) и MAX_PRIORITY (10).

Всякий раз, когда у планировщика потіков платформы есть возможнасть выбрать новый потік, он отдает предпочитання потікам с болїї высоким приоритетом. Однак приоритеты потіков сильно зависят от системы. Поскольку виртуальная машина Java опирается на реализацию потіков текущей платформы, приоритеты потіков Java сопоставляются с уровнями приоритета текущей платформы, которая может иметь больше або меньше уровней приоритета потіков.

Наприклад, в Windows имїїтся семь уровней приоритета. Некоторые приоритеты Java будуть соответствовать одному и тому же уровню операционной системы. В виртуальной машине OpenJDK для Linux приоритеты потіков вообще игнорируются — все потіки имеют одинаковый приоритет. Приоритеты потіков могли быть полезны в ранних версиях Java, которые не использовали потіки операционных систем. Вам не йде використовувати их в настоящїї час.

Для обратной совместимости все виртуальные потіки имеют приоритет NORM_PRIORITY. Так, у главного потіка програми по умолчанию приоритет - 5. Попытка змінити приоритет не имїїт ниякого эффекта.
