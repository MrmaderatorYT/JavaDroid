Атомарність та Atomics. Потікобезпека без блокувань

Останнє оновлення: 23.10.2025

В многопоточном програмировании, в том числе и на мові Java, одной из самых частых проблем є "состояние гонки" (race condition), когда декілька потіков пытаются одновременно читать и изменять одни и те же дані. Рассмотрим класический пример - простой счетчик:


public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        int numThreads = 1000;          // кількість потіков
        int incrementsPerThread = 1000; // кількість итреаций в каждом потіке

        Counter counter = new Counter();
 
        Thread[] threads = new Thread[numThreads];
        for (int i = 0; i < numThreads; i++){
             
            threads[i] = new Thread(() ->{ 

                for (int j = 0; j < incrementsPerThread; j++) {
                    counter.increment();
                }
                
            });
            threads[i].start();
        }
        // небольшая задержка в 1 секунду, чтобы потіки успели попрацювати
        Thread.sleep(1000);

        // ожидаем завершення всех потіков
        for (int i = 0; i < numThreads; i++){
            threads[i].join();
        }

        // проверяем значення счетчика
        System.out.println("Counter: " + counter.getCounter());
    }
}

class Counter{

    private volatile long counter = 0;

    long getCounter() { return counter; }
    void increment() { counter++;  }  //  НЕ АТОМАРНАЯ ОПЕРАЦИЯ!
}

Итак, здесь у нас есть клас Counter - условного счетчика, где есть змінна counter

private volatile long counter = 0;

Щоб все потіки вдели изменения цей змінної, она визначенийа с ключевым словом volatile. А в методі increment() увеличиваем на 1 значення цей змінної:

void increment() { counter++;  }

В методі main() використовуємо цей клас, создая его об'єкт:

Counter counter = new Counter();

Далїї створюємо масив из 1000 потіков, которые 1000 раз виконують метод increment() у об'єкта Counter:


Thread[] threads = new Thread[numThreads];

for (int i = 0; i < numThreads; i++){
             
    threads[i] = new Thread(() ->{ 

        for (int j = 0; j < incrementsPerThread; j++) {
            counter.increment();
        }       
    });
    threads[i].start();
}

Після завершення потіков вивідим финальное значення счетчика:

System.out.println("Counter: " + counter.getCounter());

Если мы запускуємо 1000 потіков, каждый из которых буде 1000 раз инкрементировать счетчик на единицу, то буво бы разумно ожидать, что финальное значення счетчика буде равно 1000 * 1000 = 1000 000 (то есть 1 миллион). Но посмотрим, якой на самом деле буде результат. Он может быть таким:


Counter: 944695

А может быть и таким


Counter: 870973

А может быть и совсем другим. Результат у даному випадку недетерминарован, но он может (и скорїї всего буде) отличаться от ожидаемых 1000 000. Почему так происходит? Дело в том, что операція инкремента, яку виконує в примере выше метод increment(), на самом деле она соварто из трех шагов:

  1. Читання текущего значення поля counter

  2. Увеличение прочитанного значення на 1

  3. Запис нового значення обратно в counter

И якщо два и болїї потіков виконують инкремент одновременно, то они можуть считать одно и то же значення.

Для решения цей проблемы мы могли бы використовувати блоки/функції с оператором synchronized або использоваnm блокировокb ReentrantLock. Однак оба цих способа представляютьь блокирующий механизм. И якщо потік А держит блокировку, потік Б (и В, Г, Д...) вынуждены остановиться и ждать. В системах с высокой нагрузкой це приводит к простоям потіков, частым переключениям контекста и резкому падению производительности.

Атомарность и пакет java.util.concurrent.atomic

Для решения цей завданьі язык Java надає болїї эффективный инструмент - класы из пакету java.util.concurrent.atomic, такие як AtomicInteger, AtomicBoolean, AtomicReference, AtomicLong и т.д. Эти класы використовують эффективные машинные инструкции для гарантии атомарности других операцій без використання блокувань. Операції, що надаються цими класами над одиничними змінними неблокирующие и потікобезопасные (thread-safe)

Рассмотрим застосування підобных класов на примере AtomicLong, який представляє обертку над значенням типу long с атомарными методуми для его изменения. Для створенняя об'єкта застосовується один из конструкторов класа:


AtomicLong()
AtomicLong(long initialValue)

Первая версия ініціалізує об'єкт числом 0. Второй конструктор приймає начальное значення для AtomicLong.

Ключевые методи AtomicLong:

  • long get(): повертає текущїї значення (аналогично чтению volatile long)

  • void set(long newValue): встановлює новое значення (аналогично записи в volatile long)

И також клас надає ряд атомарных операцій, которые собственно и представляютьь суть класа:

  • long incrementAndGet(): атомарно увеличивает на 1 и повертає новое значення (префиксный инкремент, ++i)

  • long getAndIncrement(): атомарно увеличивает на 1 и повертає старое значення (постфиксный инкремент, i++)

  • long decrementAndGet(): атомарно уменьшает на 1 и повертає новое (--i)

  • long getAndDecrement(): атомарно уменьшает на 1 и повертає старое (i--)

  • long addAndGet(long delta): атомарно додає delta и повертає новое (i += delta)

  • long getAndAdd(long delta): атомарно додає delta и повертає старое

  • boolean compareAndSet(long expect, long update): пытается установить значення update, лише якщо текущїї значення равно expect. Повертає true в случае успеха

Для применения атомарности и AtomicLong изменим предыдущий пример наступним чином:


import java.util.concurrent.atomic.AtomicLong;

public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        int numThreads = 1000;
        int incrementsPerThread = 1000;

        Counter counter = new Counter();
 
        Thread[] threads = new Thread[numThreads];
        for (int i = 0; i < numThreads; i++){
             
            threads[i] = new Thread(() ->{ 

                for (int j = 0; j < incrementsPerThread; j++) {
                    counter.increment();
                }
            });
            threads[i].start();
        }
        // небольшая задержка, чтобы потіки успели попрацювати
        Thread.sleep(1000);

        // ожидаем завершення всех потіков
        for (int i = 0; i < numThreads; i++){
            threads[i].join();
        }

        // проверяем значення счетчика
        System.out.println("Counter: " + counter.getCounter());
    }
}

class Counter{
    // Правильный, потікобезопасный счетчик
    private AtomicLong counter = new AtomicLong(0);

    long getCounter() { return counter.get(); }
    void increment() { counter.getAndIncrement();  }   //  АТОМАРНАЯ ОПЕРАЦИЯ!
}

Здесь изменен лише код класа Counter - вместо типу long в качестве типу змінної застосовується клас AtomicLong. Для створенняя об'єкта цього класа застосовується конструктор, в який передаєся начальное значення змінної:

AtomicLong counter = new AtomicLong(0);

То есть по умолчанию змінна counter равна 0.

В методі getCounter() за допомогою методу get() повертаємо текущїї значення счетчика:

long getCounter() { return counter.get(); }

А в методі increment() за допомогою вызова getAndIncrement() увеличиваем значення счетчика на 1:

void increment() { counter.getAndIncrement();  }

Таким чином, при запуску потіков и выполнении методу increment:


for (int i = 0; i < numThreads; i++){
             
    threads[i] = new Thread(() ->{ 

        for (int j = 0; j < incrementsPerThread; j++) {
            counter.increment();  // Выполняем атомарную операцію
        }
    });
    threads[i].start();
}

буде выполняться атомарная операція, даже якщо декілька потіков підряд попробуют вызвать метод increment()

И теперь консольный вивід буде детерминирован и собственно тем, яким и повинен быть:


Counter: 1000000

Как видно из примера, AtomicLong дает корректный результат, в то час як volatile long теряет инкременты из-за состояния гонки.

LongAdder

Атомарные типи, наприклад, тот же AtomicLong, дуже удобно використовувати для створенняя и управления счетчиками производительности, для генерации идентфиикаторов. Но йде учитывать, что при наличии дуже большого количества потіков, которые обращаются к одним и тем же атомарным значенням, производительность может снижаться. Для увеличения производительности можна використовувати класы LongAdder и LongAccumulator.

Наприклад, клас LongAdder використовується для сценария исключительно підсчета, когда не потрібно миттєво знать точное значення, а важен лише итоговый результат (сумма всех значений). Це эффективно в распространtнной ситуации, когда значення суммы не требуется до завершення всей работы. Повышение в цем случае производительности может быть существенным.

Під капотом LongAdder хранит не одно значення, а масив значений (ячїїк), общая сумма которых є текущим значенням. Каждый потік работает со своей ячейкой, и коллизии (конкуренция) возникают гораздо реже. А при вызове методу sum() LongAdder просто складывает значення всех ячїїк.

Вкратце пробежимся по функционалу класа LongAdder:

  • void add(long x): додає заданное значення.

  • void decrement(): эквивалентно add(-1).

  • double doubleValue(): повертає sum() як значення типу double після расширяющего преобразования.

  • float floatValue(): Повертає sum() як значення типу float після расширяющего преобразования.

  • void increment(): эквивалентно add(1).

  • int intValue(): повертає sum() як значення типу int після сужающего преобразования.

  • long longValue(): эквивалентно sum().

  • void reset(): сбрасывает змінні, сохраняя сумму равной нулю.

  • long sum(): повертає текущую сумму.

  • long sumThenReset(): эквивалентно sum(), за которым йде reset().

Так, в примере выше не требовалось получать промежуточное значення після каждого инкремента, поцему мы можемо заменить AtomicLong на LongAdder:


import java.util.concurrent.atomic.LongAdder;

public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        int numThreads = 1000;
        int incrementsPerThread = 1000;

        Counter counter = new Counter();
 
        Thread[] threads = new Thread[numThreads];
        for (int i = 0; i < numThreads; i++){
             
            threads[i] = new Thread(() ->{ 

                for (int j = 0; j < incrementsPerThread; j++) {
                    counter.increment();
                }
            });
            threads[i].start();
        }
        // небольшая задержка, чтобы потіки успели попрацювати
        Thread.sleep(1000);

        // ожидаем завершення всех потіков
        for (int i = 0; i < numThreads; i++){
            threads[i].join();
        }

        // проверяем значення счетчика
        System.out.println("Counter: " + counter.getCounter());
    }
}

class Counter{

    private LongAdder counter = new LongAdder();

    long getCounter() { return counter.longValue(); }   // повертаємо сумму всех значений (инкрементов) потіков
    void increment() { counter.increment();  }   //  увеличиваем на 1
}

Фактически здесь все то же самое, лише вместо AtomicLong для визначенийия счетчика застосовується LongAdder. За допомогою методу longValue() повертаємо значення счетчика (по сути сумму значений всех потіков). А за допомогою методу increment() каждый потік буде увеличивать свое значення в счетчике.

А консольный вивід також буде детерминирован, яким и повинен быть:


Counter: 1000000
