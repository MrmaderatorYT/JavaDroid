Атомарність та Atomics. Потокобезпека без блокувань

Останнє оновлення: 23.10.2025

В многопоточном программировании, в том числе и на мові Java, одной из самых частых проблем є "состояние гонки" (race condition), когда декілька потоков пытаются одновременно читать и изменять одни и те же дані. Рассмотрим классический пример - простой счетчик:


public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        int numThreads = 1000;          // количество потоков
        int incrementsPerThread = 1000; // количество итреаций в каждом потоке

        Counter counter = new Counter();
 
        Thread[] threads = new Thread[numThreads];
        for (int i = 0; i < numThreads; i++){
             
            threads[i] = new Thread(() ->{ 

                for (int j = 0; j < incrementsPerThread; j++) {
                    counter.increment();
                }
                
            });
            threads[i].start();
        }
        // небольшая задержка в 1 секунду, чтобы потоки успели поработать
        Thread.sleep(1000);

        // ожидаем завершения всех потоков
        for (int i = 0; i < numThreads; i++){
            threads[i].join();
        }

        // проверяем значение счетчика
        System.out.println("Counter: " + counter.getCounter());
    }
}

class Counter{

    private volatile long counter = 0;

    long getCounter() { return counter; }
    void increment() { counter++;  }  //  НЕ АТОМАРНАЯ ОПЕРАЦИЯ!
}

Итак, здесь у нас есть класс Counter - условного счетчика, где есть переменная counter

private volatile long counter = 0;

Щоб все потоки вдели изменения цей змінної, она определена с ключевым словом volatile. А в методе increment() увеличиваем на 1 значение цей змінної:

void increment() { counter++;  }

В методе main() используем цей класс, создая его об'єкт:

Counter counter = new Counter();

Далїї создаем массив из 1000 потоков, которые 1000 раз выполняют метод increment() у об'єкта Counter:


Thread[] threads = new Thread[numThreads];

for (int i = 0; i < numThreads; i++){
             
    threads[i] = new Thread(() ->{ 

        for (int j = 0; j < incrementsPerThread; j++) {
            counter.increment();
        }       
    });
    threads[i].start();
}

После завершения потоков выводим финальное значение счетчика:

System.out.println("Counter: " + counter.getCounter());

Если мы запускаем 1000 потоков, каждый из которых буде 1000 раз инкрементировать счетчик на единицу, то буво бы разумно ожидать, что финальное значение счетчика буде равно 1000 * 1000 = 1000 000 (то есть 1 миллион). Но посмотрим, якой на самом деле буде результат. Он может быть таким:


Counter: 944695

А может быть и таким


Counter: 870973

А может быть и совсем другим. Результат в данном случае недетерминарован, но он может (и скорїї всего буде) отличаться от ожидаемых 1000 000. Почему так происходит? Дело в том, что операция инкремента, яку выполняет в примере выше метод increment(), на самом деле она соварто из трех шагов:

  1. Чтение текущего значения поля counter

  2. Увеличение прочитанного значения на 1

  3. Запись нового значения обратно в counter

И якщо два и болїї потоков выполняют инкремент одновременно, то они можуть считать одно и то же значение.

Для решения цей проблемы мы могли бы використовувати блоки/функції с оператором synchronized або использоваnm блокировокb ReentrantLock. Однак оба цих способа представляють блокирующий механизм. И якщо поток А держит блокировку, поток Б (и В, Г, Д...) вынуждены остановиться и ждать. В системах с высокой нагрузкой це приводит к простоям потоков, частым переключениям контекста и резкому падению производительности.

Атомарность и пакет java.util.concurrent.atomic

Для решения цей задачи язык Java надає болїї эффективный инструмент - классы из пакета java.util.concurrent.atomic, такие як AtomicInteger, AtomicBoolean, AtomicReference, AtomicLong и т.д. Эти классы используют эффективные машинные инструкции для гарантии атомарности других операций без использования блокировок. Предоставляемые цими классами операции над одиночными переменными неблокирующие и потокобезопасные (thread-safe)

Рассмотрим применение подобных классов на примере AtomicLong, який представляє обертку над значением типу long с атомарными методами для его изменения. Для создания об'єкта застосовується один из конструкторов класса:


AtomicLong()
AtomicLong(long initialValue)

Первая версия ініціалізує об'єкт числом 0. Второй конструктор приймає начальное значение для AtomicLong.

Ключевые методы AtomicLong:

  • long get(): повертає текущїї значення (аналогично чтению volatile long)

  • void set(long newValue): устанавливает новое значение (аналогично записи в volatile long)

И також класс надає ряд атомарных операций, которые собственно и представляють суть класса:

  • long incrementAndGet(): атомарно увеличивает на 1 и повертає новое значение (префиксный инкремент, ++i)

  • long getAndIncrement(): атомарно увеличивает на 1 и повертає старое значение (постфиксный инкремент, i++)

  • long decrementAndGet(): атомарно уменьшает на 1 и повертає новое (--i)

  • long getAndDecrement(): атомарно уменьшает на 1 и повертає старое (i--)

  • long addAndGet(long delta): атомарно додає delta и повертає новое (i += delta)

  • long getAndAdd(long delta): атомарно додає delta и повертає старое

  • boolean compareAndSet(long expect, long update): пытается установить значение update, лише якщо текущїї значення равно expect. Повертає true в случае успеха

Для применения атомарности и AtomicLong изменим предыдущий пример наступним чином:


import java.util.concurrent.atomic.AtomicLong;

public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        int numThreads = 1000;
        int incrementsPerThread = 1000;

        Counter counter = new Counter();
 
        Thread[] threads = new Thread[numThreads];
        for (int i = 0; i < numThreads; i++){
             
            threads[i] = new Thread(() ->{ 

                for (int j = 0; j < incrementsPerThread; j++) {
                    counter.increment();
                }
            });
            threads[i].start();
        }
        // небольшая задержка, чтобы потоки успели поработать
        Thread.sleep(1000);

        // ожидаем завершения всех потоков
        for (int i = 0; i < numThreads; i++){
            threads[i].join();
        }

        // проверяем значение счетчика
        System.out.println("Counter: " + counter.getCounter());
    }
}

class Counter{
    // Правильный, потокобезопасный счетчик
    private AtomicLong counter = new AtomicLong(0);

    long getCounter() { return counter.get(); }
    void increment() { counter.getAndIncrement();  }   //  АТОМАРНАЯ ОПЕРАЦИЯ!
}

Здесь изменен лише код класса Counter - вместо типу long в качестве типу змінної застосовується класс AtomicLong. Для создания об'єкта цього класса застосовується конструктор, в який передаєся начальное значение змінної:

AtomicLong counter = new AtomicLong(0);

То есть по умолчанию переменная counter равна 0.

В методе getCounter() за допомогою метода get() возвращаем текущїї значення счетчика:

long getCounter() { return counter.get(); }

А в методе increment() за допомогою вызова getAndIncrement() увеличиваем значение счетчика на 1:

void increment() { counter.getAndIncrement();  }

Таким чином, при запуске потоков и выполнении метода increment:


for (int i = 0; i < numThreads; i++){
             
    threads[i] = new Thread(() ->{ 

        for (int j = 0; j < incrementsPerThread; j++) {
            counter.increment();  // Выполняем атомарную операцию
        }
    });
    threads[i].start();
}

буде выполняться атомарная операция, даже якщо декілька потоков подряд попробуют вызвать метод increment()

И теперь консольный вывод буде детерминирован и собственно тем, яким и повинен быть:


Counter: 1000000

Как видно из примера, AtomicLong дает корректный результат, в то час як volatile long теряет инкременты из-за состояния гонки.

LongAdder

Атомарные типы, наприклад, тот же AtomicLong, очень удобно використовувати для создания и управления счетчиками производительности, для генерации идентфиикаторов. Но йде учитывать, что при наличии очень большого количества потоков, которые обращаются к одним и тем же атомарным значениям, производительность может снижаться. Для увеличения производительности можна використовувати классы LongAdder и LongAccumulator.

Наприклад, класс LongAdder використовується для сценария исключительно подсчета, когда не потрібно мгновенно знать точное значение, а важен лише итоговый результат (сумма всех значений). Це эффективно в распространtнной ситуации, когда значение суммы не требуется до завершения всей работы. Повышение в цем случае производительности может быть существенным.

Под капотом LongAdder хранит не одно значение, а массив значений (ячїїк), общая сумма которых є текущим значением. Каждый поток работает со своей ячейкой, и коллизии (конкуренция) возникают гораздо реже. А при вызове метода sum() LongAdder просто складывает значения всех ячїїк.

Вкратце пробежимся по функционалу класса LongAdder:

  • void add(long x): додає заданное значение.

  • void decrement(): эквивалентно add(-1).

  • double doubleValue(): повертає sum() як значение типу double после расширяющего преобразования.

  • float floatValue(): Повертає sum() як значение типу float после расширяющего преобразования.

  • void increment(): эквивалентно add(1).

  • int intValue(): повертає sum() як значение типу int после сужающего преобразования.

  • long longValue(): эквивалентно sum().

  • void reset(): сбрасывает переменные, сохраняя сумму равной нулю.

  • long sum(): повертає текущую сумму.

  • long sumThenReset(): эквивалентно sum(), за которым йде reset().

Так, в примере выше не требовалось получать промежуточное значение после каждого инкремента, поцему мы можемо заменить AtomicLong на LongAdder:


import java.util.concurrent.atomic.LongAdder;

public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        int numThreads = 1000;
        int incrementsPerThread = 1000;

        Counter counter = new Counter();
 
        Thread[] threads = new Thread[numThreads];
        for (int i = 0; i < numThreads; i++){
             
            threads[i] = new Thread(() ->{ 

                for (int j = 0; j < incrementsPerThread; j++) {
                    counter.increment();
                }
            });
            threads[i].start();
        }
        // небольшая задержка, чтобы потоки успели поработать
        Thread.sleep(1000);

        // ожидаем завершения всех потоков
        for (int i = 0; i < numThreads; i++){
            threads[i].join();
        }

        // проверяем значение счетчика
        System.out.println("Counter: " + counter.getCounter());
    }
}

class Counter{

    private LongAdder counter = new LongAdder();

    long getCounter() { return counter.longValue(); }   // возвращаем сумму всех значений (инкрементов) потоков
    void increment() { counter.increment();  }   //  увеличиваем на 1
}

Фактически здесь все то же самое, лише вместо AtomicLong для определения счетчика застосовується LongAdder. С помощью метода longValue() возвращаем значение счетчика (по сути сумму значений всех потоков). А за допомогою метода increment() каждый поток буде увеличивать свое значение в счетчике.

А консольный вывод також буде детерминирован, яким и повинен быть:


Counter: 1000000
