CompletableFuture та проміси. Обробка результату асинхронних завдань

Останнє оновлення: 29.10.2025

Об'єкт Future, який представляє асинхронную завданьу, дозволяє отримати результат завданьі за допомогою методу get(), який приведет к блокировки текущего потіка до тех пор, поки результат асинхронной завданьі не станет доступен. Клас CompletableFuture, який реалізує інтерфейс Future, надає альтернативний механизм для получения результата. CompletableFuture використовує функцію обратного вызова, которая вызвается с результатом асинхронной завданьі, когда он станет доступен. Можна сказать, что CompletableFuture - це Future, який может быть явно завершен (с установкой его значення и статуса)

Крім того, CompletableFuture реалізує інтерфейс CompletionStage, який представляє этап асинхронного вычисления и виконує действие або вычисляет значення після завершення другого этапа. Можна сказать, что CompletableFuture дозволяє реализовать нечто похожїї на промисы (promise) в некоторых языках програмирования (як, наприклад, в JavaScript)

Створення CompletableFuture

Для непосредственного створенняя об'єкта CompletableFuture можна використовувати конструктор без параметрів:

CompletableFuture future = new CompletableFuture();

Клас CompletableFuture<T> є обобщенным и типизируется типом результата. Так, в коді выше предполагается, что результатом асинхронной завданьі буде значення типу Integer.

Після створенняя об'єкта за допомогою конструктора его можна настроить на виконання якой-або завданьі. Однак клас CompletableFuture також представляє ряд методів, которые дозволяють створити об'єкт данного класа с вже добавленной асинхронной завданьей, яку йде выполнить. Наприклад, це статический метод CompletableFuture<U>.supplyAsync():


static <U> CompletableFuture<U> supplyAsync(Supplier<U> supplier)
static <U> CompletableFuture<U> supplyAsync(Supplier<U> supplier, Executor executor)

Цет метод имїїт две версии. Обе версии в качестве результата повертає новый об'єкт CompletableFuture. И обе цих версии в качестве первого параметра приймають об'єкт Supplier, який и представляє выполняемое действие. Данный інтерфейс є функциональным и имїїт один метод get(), який повертає некоторое значення


interface Supplier<T>{
    T get();
}

Результат методу get() затем передаєся в обратный вызов - асинхронную завданьу, которая и обробляє данный результат и которая встановлюєся за допомогою дополнительных методів.

Различие між цими обеими версиями соварто лишь в том, як именно виконується асинхронная завданьа. Під второй версии вторым параметром передаєся об'єкт Executor, який и виконує завданьу. Для первой же версии асинхронная завданьа виконується за допомогою потіка из пула ForkJoinPool (ForkJoinPool представляє специализированный пул потіков, разработанный для параллельного програмирования). То есть при выполнении завдань вы можете положиться на ForkJoinPool, або для большего контроля предоставить свой об'єкт Executor, який выполнит завданьу.

Рассмотрим на простейшем примере вычисления факториала:


int number = 5;  // исходное число для вычисления факториала

CompletableFuture<Integer> future = CompletableFuture.supplyAsync(() -> {
 
    // якщо число менше 1, генерируем виняток
    if(number < 1) throw new RuntimeException("Number must be greater than 0");

    int n = 1;
    int result = 1;
    while(n <= number) result *= n++; 
    return result;  // повертаємо результат завданьі
});

У даному випадку в метод CompletableFuture.supplyAsync() неявно передаєся об'єкт Supplier в виде лямбда-выражения. Це лямбда-выражение берет внешнюю змінну number и вычисляет її факториал и повертає результат - це и буде результат методу get(). Конечно, можна буво бы и явным чином визначити об'єкт Supplier:


int number = 5;  // исходное число для вычисления факториала

Supplier<Integer> task = () -> {
 
    // якщо число менше 1, генерируем виняток
    if(number < 1) throw new RuntimeException("Number must be greater than 0");

    int n = 1;
    int result = 1;
    while(n <= number) result *= n++;
    return result;  // повертаємо результат завданьі
};
CompletableFuture<Integer> future = CompletableFuture.supplyAsync(task);

Зверніть увагу, что функція, которая представляє інтерфейс Supplier, не может генерировать проверяемое виняток (checked excetion). Тому в примере выше для генерации винятки при передаче некорректного числа використовується тип RuntimeException.

Додавання обратного вызова

Після створенняя CompletableFuture и установки выполняемой завданьі нам треба добавить обратный вызов, який буде вызываться при завершении CompletableFuture. Для цього в класе визначений большой набор методів. Приведу лишь некоторые:

  • CompletableFuture<Void> thenAccept(Consumer<? super T> action)

    Застосовує функцію к результату и повертає void.

  • <U> CompletableFuture<U> thenApply(Function<? super T, ? extends U> fn)

    Застосовує функцію к результату.

  • CompletableFuture<Void> thenRun(Runnable action)

    Виконує Runnable с результатом типу void.

  • <U> CompletableFuture<U> thenCompose(Function<? super T, ? extends CompletionStage<U>> fn)

    Вызывает функцію для результата и виконує повертаємоый об'єкт Future.

Наприклад, використовуємо метод thenAccept(), який приймає об'єкт інтерфейса Consumer. Цет інтерфейс имїїт метод accept(), який приймає некоторое значення:

void accept(T t)

Таким чином, обратный вызов повинен представлять функцію, которая приймає значення произвольного типу, а в качестве повертаємоого типу испольует void

Рассмотрим на примере факториала:


import java.util.concurrent.*;
import java.util.function.Supplier;
 
class Program{
 
    public static void main(String[] args) throws Exception{
          
        System.out.println("Main thread started...");
 
        int number = 5;  // исходное число для вычисления факториала
       
        // определяем завданьу, которая вычисляет факториал
        Supplier<Integer> task = () -> {

            if(number < 1) throw new RuntimeException("Number must be greater than 0");
            int n = 1;
            int result = 1;
            while(n <= number) {

                result *= n++;
            }
            return result;  // повертаємо результат завданьі
        };
 
        CompletableFuture<Integer> future = CompletableFuture.supplyAsync(task);

        // передаємо в thenAccept обратный вызов
        future.thenAccept(result -> System.out.printf("factorial of %d is %d\n", number, result));

        // future.thenAccept - не блокує основной потік 
        // и одновременно мы можемо выполнять в основном потіке деяку работу 
        System.out.println("Main thread works...");
 
        Thread.sleep(2000);

        System.out.println("Main thread finished...");
    }
}

Рассмотрим на передачу обратного вызова:

future.thenAccept(result -> System.out.printf("factorial of %d is %d\n", number, result));

Здесь в метод передаєся лямбда-выражение, в котором параметр result представляє результ асинхронной завданьі - результат вычисления факториала. Тело лямбда-выражения же представляє вивід результата вычисления.

Если суммировать все по этапам, то програма буде выполняться наступним чином:

  • System.out.println("Main thread started...")

    Основной потік (main) запускуєся и виводить це сообщение.

  • CompletableFuture future = CompletableFuture.supplyAsync(task);

    Передає раніше визначену завдання task (обчислення факторіалу) в общий пул потіков (по умолчанию ForkJoinPool.commonPool()). Эта завданьа начинает выполняться параллельно в другом потіке.

  • future.thenAcceptAsync(result -> System.out.printf("factorial of %d is %d\n", number, result))

    Регистрирует обратный вызов (колбэк). Цет колбэк не виконується немедленно. Он ждет, поки завданьа task не завершится и не возвратит неокторый результат.

  • System.out.println("Main thread works...")

    Основной потік не ждет завершення завданьі task, а немедленно продолжает работу (у даному випадку для имитации работы вивідим сообщение)

  • Виконання завданьі

    В це час (main або друкує "Main thread works...", або вже спит) параллельный потік дуже быстро вычисляет факториал 5 (це 120). Це занимает доли миллисекунды.

  • Виконання колбэка

    Сразу після завершення завданьі task запускуєся зарегистрированный за допомогою функції thenAccept обратный вызов (обычно в том же потіке из пула) и виводить: factorial of 5 is 120.

  • Thread.sleep(2000);

    Основной потік "спит" 2000 мс. К цему моменту колбэк почти наверняка вже отработал и вывел результат.

  • System.out.println("Main thread finished...")

    Через 2 секунды основной потік просыпается и виводить сообщение, після чего програма завершуєся.

    • Наиболїї вероятный порядок вивіда буде таким:

    
Main thread started...
Main thread works...
factorial of 5 is 120
Main thread finished...

    Таким чином, можна обробляти результат, як лише он станет доступен, не блокируя основой потік.

    Но у даному випадку варто відзначити, что, поскольку потіки, використовуємоые CompletableFuture по умолчанию (из пула ForkJoinPool.commonPool()), є потіками-демонами (daemon threads) - по сути фоновыми потіками, то JVM (Виртуальная машина Java) не ждет, поки они завершатся. JVM работает до тех пор, поки жив хотя бы один потік, який не є "демоном" . В програме выше единственный потік, який не є демоном - це потік методу main. Тому якщо убрать Thread.sleep, то потік main дуже быстро выполнит методи supplyAsync() и thenAccept(), надрукує "Main thread works..." и "Main thread finished..." и немедленно завершится.

    Как лише main завершуєся, JVM видит, что активных потіков-не-демонов не осталось, и принудительно завершує всю програму. И в цем случае асинхронная завданьа task и її колбэк в thenAccept() просто не успевают выполниться до завершення основного потіка. Тому именно Thread.sleep(2000) в коді выше "удерживает" основной потік живым, давая асинхронной завданье достатньо времени, чтобы выполниться и напечатать результат.

    Вместо использования задержек мы могли бы явным чином визначити об'єкт Executor, який буде выполнять завданьі. Наприклад:

    
import java.util.concurrent.*;
import java.util.function.Supplier;
 
class Program{
 
    public static void main(String[] args) throws Exception{
          
        System.out.println("Main thread started...");
 
        int number = 5;  // исходное число для вычисления факториала
       
        // определяем завданьу, которая вычисляет факториал
        Supplier<Integer> task = () -> {

            if(number < 1) throw new RuntimeException("Number must be greater than 0");
            int n = 1;
            int result = 1;
            while(n <= number)  result *= n++; 

            // имитируем долгую работу
            try {Thread.sleep(2000);}
            catch(InterruptedException _){}

            return result;  // повертаємо результат завданьі
        };

        // определяем об'єкт Executor, який буде выполнять завданьу
        ExecutorService executor = Executors.newCachedThreadPool();
        CompletableFuture<Integer> future = CompletableFuture.supplyAsync(task, executor);
        future.thenAccept(result -> System.out.printf("factorial of %d is %d\n", number, result));

        // future.thenAccept - не блокує основной потік 
        // и одновременно мы можемо выполнять в основном потіке деяку работу 
        System.out.println("Main thread works...");
        System.out.println("Main thread finished...");
        // закрываем исполнителя
        executor.close();  
    }
}

    В принципе здесь виконуються те же самые действия, лише теперь я убрал задержку для основного потіка и добавил аналогичную задержку в завданьу по вычислению факториала, чтобы эта завданьа завершалась після всех действий в основном потіке методу main.

    И також добавлено створення об'єкта Executor, який передаєся в метод supplyAsync():

    
ExecutorService executor = Executors.newCachedThreadPool();
CompletableFuture<Integer> future = CompletableFuture.supplyAsync(task, executor);

    У даному випадку создаваемый об'єкт Executor буде використовувати один из платформенных потіков из пула. Причем далїї цей Executor буде выполнять як завданьу по вычислению факториала, так и коллбек, який виводить результат на консоль.

    В итоге мы скорїї всего получим следующий консольный вивід:

    
Main thread started...
Main thread works...
Main thread finished...
factorial of 5 is 120

    Поскольку в асинхронной завданье действует 2-секундная задержка, то скорїї всего асинхронная завданьа факториала буде выполнена після вивіда на консоль рядки "Main thread finished..." в методі main.

    Створення цепочки обратных вызовов

    Аналогичным чином мы можемо застосовувати и другие методи CompletedFuture для добавления обратных вызовов для обработки результатів. Болїї того мы можемо обробляти результати других обратных вызовов. Наприклад, возьмем метод thenApply():

    <U> CompletableFuture<U> thenApply(Function<? super T, ? extends R> fn)

    Параметром методу є об'єкт інтерфейса Function - по сути це некоторая функція, которая приймає об'єкт типу T и повертає об'єкт типу R, то есть преобразует результат типу T в об'єкт типу R. Або якщо быть точнїї функція обратного вызова повинна соответствовать методу apply() інтерфейса Function:

    R apply(T t)

    Допустим, після вычисления факториала нам треба увеличить его в 2 раза:

    
import java.util.concurrent.*;
import java.util.function.Supplier;
import java.util.function.Function;
import java.util.function.Consumer;
 
class Program{
 
    public static void main(String[] args) throws Exception{
          
        System.out.println("Main thread started...");
 
        int number = 5;  // исходное число для вычисления факториала
       
        // определяем завданьу, которая вычисляет факториал
        Supplier<Integer> factorialTask = () -> {

            if(number < 1) throw new RuntimeException("Number must be greater than 0");
            int n = 1;
            int result = 1;
            while(n <= number)  result *= n++; 
            return result; 
        };
        // определяем завданьу, которая увеличивает в два раза
        Function<Integer, Integer> doubleTask = result ->  result * 2;
        // определяем завданьу, которая виводить на консоль конечный результат
        Consumer<Integer> printTask = result -> System.out.printf("Final result: %d\n", result);

        // определяем об'єкт Executor, який буде выполнять завданьі
        ExecutorService executor = Executors.newCachedThreadPool();
        // определяем коллбеки
        CompletableFuture
                .supplyAsync(factorialTask, executor)
                .thenApply(doubleTask)
                .thenAccept(printTask);

        System.out.println("Main thread works...");
        System.out.println("Main thread finished...");
        
        executor.close();  // закрываем исполнителя
    }
}

    Ключевой момент програми - це, конечно, визначенийие цепочки выполняемых завдань:

    
CompletableFuture
    .supplyAsync(factorialTask, executor)
    .thenApply(doubleTask)
    .thenAccept(printTask);

    Рассмотрим поэтапно:

    1. supplyAsync(factorialTask, executor)

      Визначає асинхронную завданьу, которая виконується - це завданьа factorialTask, которая вычисляет факториала. Эта завданьа представляє тип Supplier<Integer> и поцему повертає значення типу Integer

    2. thenApply(doubleTask)

      Застосовує к результату предыдущей завданьі (в нашем случае це обчислення факторіалу) функцію doubleTask. Эта завданьа представляє тип Function<Integer, Integer>. То есть функція отримує значення типу Integer (факториал числа - результат завданьі factorialTask) и повертає також значення типу Integer - удвоенное значення факториала.

    3. thenAccept(printTask)

      Застосовує к результату предыдущей завданьі (в нашем случае це результат завданьі doubleTask) функцію printTask, которая представляє тип Consumer<Integer>. Эта завданьа отримує значення типу Integer (результат завданьі doubleTask) и ничего не повертає (фактически повертає void) и просто виводить результат на консоль.

    Консольный вивід програми:

    
Main thread started...
Main thread works...
Main thread finished...
Final result: 240
