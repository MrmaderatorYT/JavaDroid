Java Native Interface (JNI)

Останнє оновлення: 21.12.2025

Мы можемо написать всю программу целиком на Java, учитывая большие возможнасти языка, большое количество досьупных фреймворков и библиотек на цем языке. Но бывают ситуации, когда может потребоваться одновременно вместе с кодом на Java використовувати також код на другом языке, в частности, на C/C++. Наприклад, якщо приложению требуется доступ к системным функциям або устройствам, недоступным через платформу Java, або якщо вже имїїтся значний обсяг протестованого и відлагодженого кода на другом языке, який треба інтегрувати в программу на мові Java. Тому розглянемо як мы можемо взаємодіяти в программе на Java с нативным кодом на C/C++

Начиная вже с версии Java 1.1 платформа Java имїїт специальный API для взаимодействия с нативным кодом C, який називається Java Native Interface (JNI). Начиная с версии Java 20 був добавлен новый, болїї простой и удобный API - Foreign Functions and Memory API (FFM API). Тем не менїї JNI по прежнему может використовуватися, особенно на старых версиях платформы. Тому вначале вкратце розглянемо, базовые моменты JNI

Общий процесс связывания нативного метода с Java-программой за допомогою JNI выглядит наступним чином:

  1. Оголошуємо нативний метод-обгортку для функції Си в Java-классе.

  2. С помощью команды javac -h генерируем заголовочный файл с оголошенням метода на языке C.

  3. Определяем код для нативного метода на языке C.

  4. Компилируем код на Си в разделяемую библиотеку.

  5. Загружаем эту библиотеку в программе на Java

Рассмотрим по шагово весь процесс.

Определение обертки функції на Java

При использовании JNI для функції языка C/C++, яку мы хочемо використовувати, необхідно оголосити проксі-метод за допомогою ключевого слова native. Це ключевое слово повідомляє компилятору, что метод буде определен зовні. Нативные методы не містять кода Java, и за заголовком метода відразу йде крапка с комою. Тому оголошення нативных методів выглядят аналогично оголошенням абстрактных методів.

Предположим, есть функция на языке C, яку необхідно використовувати в коде на мові Java. Для наглядности возьмем стандартную библиотечную функцию printf с одним аргументом и для її применения определим файл HelloNative.java со наступним кодом на мові Java:


class HelloNative {  
    public static native int printf(String str);
}

Здесь определен класс HelloNative с одним нативным методів, имя которого соответствует названию функції printf. Нативные методы можуть быть як статическими, так и нестатическими. В данном случае нативний метод також оголошений як статический.

Далїї нам треба определить заголовочный файл на языке С. Для цього необхідно скомпилироть данный класс с флагом -h, указав каталог, в який повинні быть помщены заголовочные файлы:

javac -h . HelloNative.java

В итоге буде скомпилирован файл HelloNative.h со наступним кодом:

/* DO NOT EDIT THIS FILE - it is machine generated */
#include <jni.h>
/* Header for class HelloNative */

#ifndef _Included_HelloNative
#define _Included_HelloNative
#ifdef __cplusplus
extern "C" {
#endif
/*
 * Class:     HelloNative
 * Method:    printf
 * Signature: (Ljava/lang/String;)I
 */
JNIEXPORT jint JNICALL Java_HelloNative_printf
  (JNIEnv *, jclass, jstring);

#ifdef __cplusplus
}
#endif
#endif

Данный файл містить оголошення функції Java_HelloNative_printf. Причем обратите внимание, як називається функция: "Java_HelloNative_printf", а не просто "printf". В JNI имена функций строго регламентированы: сначала а додаєся префикс "Java", затем додаєся пакет класса (якщо пакет вложенный, наприклад, "com..test", то в полном названии пакета крапка заменяется на прочерк - "com__test") и в конце додаєся название класса и название функції через подчеркивание. То есть отримуєся: Java_ + ИмяКласса_ + ИмяМетода. Значит, в Java цей метод оголошений в классе HelloNative як native int printf(String s)

В принципе мы могли бы и вручную прописать подобный файл, но всегда есть вероятность ошибиться, поцему лучше використовувати автогенерацию.

Определение кода на Си

Далїї определим ще один файл - HelloNative.c со наступним кодом на языке С:

#include "HelloNative.h"
#include <stdio.h>

JNIEXPORT jint JNICALL Java_HelloNative_printf (JNIEnv *env, jclass cl, jstring jstr) {

    const char *cstr = (*env)->GetStringUTFChars(env, jstr, NULL);
    int numchars = printf(cstr);
    (*env)->ReleaseStringUTFChars(env, jstr, cstr);
    return numchars;
}

Рассмотрим подробно, что происходит в каждой строке:

  1. #include "HelloNative.h

    Прежде всего подключаем наш автосгенерированный заголовочный файл, чтобы Java смогла определить функцию при її вызове.

  2. #include <stdio.h>

    Подключение стандартного заголовочного файла языка С, який містить прототипы функций для работы с подсистемой ввода-вывода

  3. JNIEXPORT jint JNICALL Java_HelloNative_printf

    JNIEXPORT и JNICALL - це специальные макросы, которые гарантируют, что функция буде видна Java-машине (JVM) и буде використовувати правильное соглашение о вызовах. Между макросами указан возвращаемый тип - jint, которому в мові Java соответствует int. И после макросов идет имя функції - Java_HelloNative_printf и її определение.

  4. (JNIEnv *env, jclass cl, jstring jstr)

    Каждая нативная функция отримує параметр env, який необходим для доступа к API языка Си для нативных вызовов.

    Параметр cl - це дескриптор Java-класса (для нестатических методів цему параметру передаєся дескриптор об'єкта).

    Параметр jstr типу jstring - собственно то значение (строка), которое буде передаваться при вызове данного метода в коде Java.

  5. Далїї идет собственно код метода. И начинается он с получения строки (GetStringUTFChars)

    const char *cstr = (*env)->GetStringUTFChars(env, jstr, NULL);

    В Java строки хранятся в формате Unicode, а функції C (як printf) работают с массивами char.

    Эта строка конвертирует Java-строку (jstring) в стандартную C-строку (UTF-8).

  6. Вывод на экран

    int numchars = printf(cstr);

    Здесь вызывается стандартная функция языка C printf. Она выводит строку в консоль. Переменная numchars сохраняет количество напечатанных символов.

  7. Освобождение памяти (ReleaseStringUTFChars)

    (*env)->ReleaseStringUTFChars(env, jstr, cstr);

    Це критически важный шаг. Поскольку JNI выделил память под cstr, її необхідно освободить вручную, чтобы избежать утечек памяти.

    Виртуальная машина повинна знать, когда мы закончили використовувати строку, чтобы она могла выполнить сборку мусора. Поскольку сборщик мусора работает в отдельном потоке и может прерывать выполнение нативных методів, то необхідно вызвать функцию ReleaseStringUTFChars.

  8. Підзврат значения

    return numchars;

    Количество напечатанных символов повертаєся обратно в Java-код.

Компиляция библиотеки на Си

Итак, у нас есть заголовочный файл и файл кода C, и теперь нам треба скомпилировать весь цей нативний код на языке C в динамически загружаемую библиотеку. Конкретные подробности зависят от выбранного компилятора. В данном случае посмотрим, як це сделать, используя 2 распространенных компилятора.

Так, при использовании компилятора GCC на Linux команда компиляции буде выглядеть наступним чином:


gcc -fPIC -I $JAVA_HOME/include -I $JAVA_HOME/include/linux -shared -o libHelloNative.so HelloNative.c

Здесь варто учитывать, что переменная окрвжения $JAVA_HOME повинна быть определена и указывать на папку установки java. Соответственно путь $JAVA_HOME/include буде указывать на папку с заголовочными файлами, которые необходимы для компиляции библиотеки на C, в частности, файл jni.h, який подключался выше в одном из файлов. Тем не менїї эта переменная может быть и не установлена, в цем случае (якщо нет желания устанавливать переменную $JAVA_HOME) можна передать конретный путь к папке, где располагаются заголовочные файлы. Наприклад, якщо java локально установлена в каталоге "/usr/java/jdk-25", то соответственно нам треба прописать


gcc -fPIC -I /usr/java/jdk-25/include -I /usr/java/jdk-25/include/linux -shared -o libHelloNative.so HelloNative.c

В любом случае после виконання цей команды в текущей папке повинен появиться файл библиотеки libHelloNative.so (название самой библиотеки - "HelloNative" - без префикса "lib" и расширения файла)

Если текущая ОС - Windows и компилятор - VisualC++ от Microsoft, то там мы могли бы выполнить следующую команду


cl -I %JAVA_HOME%\include -I %JAVA_HOME%\include\win32 -LD HelloNative.c -FeHelloNative.dll

В цем случае в текущей папке появится файл "HelloNative.dll" ((название самой библиотеки також "HelloNative").

Определение кода Java

Для загрузки библиотеки нам треба добавить в код программы вызов метода System.loadLibrary, в який передаєся название библиотеки. Щоб гарантировать загрузку библиотеки виртуальной машиной до первого использования класса, можна використовувати блок статической инициализации. Наприклад, определим в текущей папке следующий файл Program.java:


class Program {
  
    public static void main(String[] args) {
        
        HelloNative.printf("Hello \n");    
    }

    static {  
        System.loadLibrary("HelloNative");
    }
}

Таким чином, в папке проекта у нас будуть следующие файлы:

  • HelloNative.java: класс с нативным методом-оберткой для функції языка Си

  • HelloNative.class: скомпилированный класс

  • HelloNative.h: автосгенерированный заголовочный файл

  • HelloNative.c: реализация функції из заголовочного файла

  • libHelloNative.so/HelloNative.dll: скомпилированная библиотека

  • Program.java: основной файл программы

Запуск программы

Скомпилируем программу на Java стандартной командой

javac Program.java

Для запуска программы прежде всего нам треба передать утилите java дополнительную опцию, которая як-бы разрешает использование нативного кода в программе на Java. Если модули не використовуються, то программа запускаєся наступним чином:

java --enable-native-access=ALL-UNNAMED ...

В модульной же программе необхідно перечислить все модули, для которых разрешен нативний доступ:

java --enable-native-access=module1, module2 ...

При запуске на Linux також необхідно добавить текущий каталог в путь к библиотекам, чтобы java нашла нашу библиотеку. Для цього можна установить переменную среды LD_LIBRARY_PATH:

export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH

Здесь за допомогою точки указываем текущую папку. Або можна використовувати системное свойство java.library.path:

java --enable-native-access=ALL-UNNAMED -Djava.library.path=. Program

Например:


eugene@Eugene:/workspace/java/test$ java --enable-native-access=ALL-UNNAMED -Djava.library.path=. Program
Hello 
eugene@Eugene:/workspace/java/test$
