Підстановочні знаки wildcards в узагальненнях

Останнє оновлення: 10.10.2025

В мові Java обобщения играют важную роль в обеспечении безопасности типов и создании гибкого кода. Однак, при работе с наследованием и обобщениями можуть возникать сложности. Именно здесь на помощь приходят wildcards (подстановочные знаки або метасимволы), предоставляя мощный инструмент для работы с неизвестными типуми.

В контексте обобщений wildcard або подстановочный знак представляє представляє неизвестный тип и обозначается за допомогою вопросительного знака ?. Он дозволяє создавать болїї гибкий функционал, який может работать с различными типуми данных.

Існує три види підстановочних знаків різних цілей:

  • Необмежений Wildcard (?)

  • Wildcard, обмежений "згори" (ограниченный базовым классом) (? extends Type)

  • Wildcard, обмежений "знизу" (ограниченный производным классом) (? super Type)

Розглянемо ці типи підстановочних знаків та сценарії їх використання з прикладу наступного класу Person:


class Person<T>{

    private T id;
    private String name;

    T getId(){ return id; }
    void setId(T id){ this.id = id; }

    String getName() { return name; }

    Person(T id, String name){

        this.id = id;
        this.name = name;
    }
}

Тип Person є обобщенный и типизирован параметром T, який надає тип для поля id.

Але передусім подивимося ситуацію, де підстановочні знаки можуть допомогти:


public class Program{
       
    public static void main(String[] args) {
           
        Person<Object> bob = new Person<>(2.0, "Bob");
        Person<Integer> tom = new Person<>(1, "Tom");
        Person<String> sam = new Person<>("F-456", "Sam");

        printPersonInfo(bob);
        printPersonInfo(tom); // ! ОШибка:  tom - НЕ представляє тип Person<Object>
        printPersonInfo(sam); // ! ОШибка:  sam - НЕ представляє тип Person<Object>
    }

    static void printPersonInfo(Person<Object> person) {

        System.out.print("Name: " + person.getName());
        Object id = person.getId();
        System.out.println(";  Id: " + id);
    }
}

При попытке компиляции цей программы мы столкнемся с ошибками. Почему? Здесь метод printPersonInfo() в качестве параметра приймає об'єкт типу Person<Object>. Типизация типом Object казалось бы охватит все возможные ситуации для поля id, в том числе когда це поле представляє типы Integer и String (як наследника класса Object). Однак в реальности мы сможемо передать в цей метод об'єкты Person, которые типизирированы типуми, отличными от Object:


Person<Integer> tom = new Person<>(1, "Tom");
Person<String> sam = new Person<>("F-456", "Sam");

printPersonInfo(tom); // ! ОШибка:  tom - НЕ представляє тип Person<Object>
printPersonInfo(sam); // ! ОШибка:  sam - НЕ представляє тип Person<Object>

І підстановочні символи wildcard дозволяють вирішити цю проблему.

Необмежений Wildcard (?)

Необмежений wildcard (Unbounded Wildcard) використовується, коли потрібно працювати з узагальненим типом, де конкретний тип невідомий чи важливий. Такий знак підстановки фактично позначає "будь-який тип". Такий тип значень підстановки може бути корисним, коли логіка вашого методу не залежить від конкретного типу, що використовується в узагальненні.

Розглянемо такий приклад:


public class Program{
       
    public static void main(String[] args) {
           
        // Пример использования:
        Person<Integer> tom = new Person<>(123, "Tom");
        Person<String> bob = new Person<>("A-456", "Bob");

        printPersonInfo(tom);   // Name: Tom;  Id: 123
        System.out.println();
        printPersonInfo(bob);   // Name: Bob;  Id: A-456
    }
    static void printPersonInfo(Person<?> person) {
        // Мы можемо безопасно вызывать методы, не связанные с дженериком
        System.out.print("Name: " + person.getName());
        
        // Мы можемо отримати id, но компилятор буде рассматривать его як Object
        Object id = person.getId();
        System.out.println(";  Id: " + id);
    }
}

В данном случае определен метод printPersonInfo, який просто выводит информацию о человеке. Обратите внимание на тип его параметра:

Person<?> person

То есть параметр person представляє класс Person, но яким типом типизирован цей класс, неважно. Поскольку здесь нам не важен тип его поля id - мы просто выводим его на консоль. В цем случае використовується строковое представление поля id. Тому в цей метод мы можемо передать об'єкты класса Person, типизированные различными типуми, наприклад, Integer або String:


Person<Integer> tom = new Person<>(123, "Tom");
Person<String> bob = new Person<>("A-456", "Bob");

printPersonInfo(tom);   // Name: Tom;  Id: 123
printPersonInfo(bob);   // Name: Bob;  Id: A-456

Но поскольку компилятор здесь не знает, якой именно тип очікує об'єкт Person<?>, мы не можемо выполнять ряд операций с полем id, наприклад, установить его за допомогою метода setId():


public class Program{
       
    public static void main(String[] args) {
           
        // Пример использования:
        Person<Integer> tom = new Person<>(123, "Tom");
        changePerson(tom, 25);
    }
    static void changePerson(Person<?> person, Object id) {

        person.setId(id); // ! ошибка
    }
}

Здесь при попытке установить id через вызов person.setId(id) мы полуим ошибку во час компиляции.

Wildcard, обмежений базовим класом (? extends Type)

Wildcard, обмежений "згори" або обмежений базовим класом, означає "будь-який тип, який представляє даний базовий клас або його похідний клас. Задається у вигляді наступного виразу:

? extends Type

Тобто тип повинен представляти похідний від Type тип або сам Type

"Зверху"? Тому що в ієрархії типів базові типи розташовуються вгорі. Object) відсікаються.

Проте ми можемоо написати так:

Наприклад, определим метод, який работает лише с теми об'єктами Person, у которых поле id представляє число, наприклад, Integer, Double. Отличительная особенность встроенных числовых типов в Java в том, что они наследуются от базового класса Number. Тому мы можемо використовувати тип Number в качестве ограничения сверху:


public class Program{
       
    public static void main(String[] args) {
           
        Person<Integer> tom = new Person<>(10, "Tom");
        Person<Double> bob = new Person<>(25.5, "Bob");

        processNumericId(tom);      // Id as double: 10.0
        processNumericId(bob);      // Id as double: 25.5
    }

    static void processNumericId(Person<? extends Number> person) {
        // Мы можемо безопасно "прочитать" id як Number
        Number id = person.getId();
        System.out.println("Id as double: " + id.doubleValue());
    }
}

Здесь метод processNumericId() приймає об'єкт Person, який типизирован типом NUmber або его наследниками:

void processNumericId(Person<? extends Number> person)

Благодаря цему компилятор внутри метода буде рассматривать поле id (которое имїїт тип T) як поле типу Number. Тому мы сможемо внутри метода работать с id як с об'єктом типу Number, наприклад, вызвать его метод doubleValue() для получения id в виде double.

Також ми не можемоо встановити id, навіть якщо використовувати об'єкт Number:


public static void main(String[] args) {
           
    Person<String> sam = new Person<>("A-456", "Sam");
    Person<Object> alice = new Person<>(123, "Alice");

    processNumericId(sam);      // ! Ошибка
    processNumericId(alice);    // ! Ошибка
}

В первом случае (processNumericId(sam)) об'єкт Person типизирован типом String, а String не входит в иерархию типу Number. Тому получаем ошибку.

Під втором случае (processNumericId(alice)) об'єкт Person типизирован типом Object. Однак в иерархии типов тип Object располагается выше типу Number (є его базовым классом). Тому також получаем ошибку.


static void setNumericId(Person<? extends Number> person, Number id) {
        
    person.setId(id);  // ! Ошибка
}

Компилятор не знает, что именно очікує об'єкт — Integer або Double, поцему не позволит установить якое-або конкретное значение.

Wildcard з обмеженням похідним типом (?super Type)

Wildcard, ограниченный "снизу" або ограниченный производным типом, означает "любой тип, який представляє данный тип або его базовый тип. Под базовым типом понимается любой тип, який расположен выше по иерархии (наприклад, для типу Integer - це Number (прямой базовый тип) и Object (непрямой базовый тип)). Задается в виде следующего выражения:

? super Type

Тобто тип повинен представляти сам Type або його базовий тип (супертип). Тому всі типи-спадкоємці від самого Type у разі відсікаються.

Допустим, нам нвжен метод для обновления id, який представлять тип Integer, або его базовый класс (Number, Object). В цем случае мы можемо установить ограничение снизу в виде типу Integer:


public class Program{
       
    public static void main(String[] args) {
           
        Person<Integer> pInt = new Person<>(1, "Tom");
        Person<Number> pNum = new Person<>(2.0, "Bob");
        Person<Object> pObj = new Person<>("F-456", "Sam");

        setIntegerId(pInt, 100); // OK
        setIntegerId(pNum, 200); // OK
        setIntegerId(pObj, 300); // OK

        // проверка
        System.out.println("pInt Id: " + pInt.getId()); // pInt Id: 100
        System.out.println("pNum Id: " + pNum.getId()); // pNum Id: 200
        System.out.println("pObj Id: " + pObj.getId()); // pObj Id: 300
    }
    static void setIntegerId(Person<? super Integer> person, int newId) {
        // Мы можемо безопасно "записать" Integer,
        // оскільки любой супертип Integer сможет его принять.
        person.setId(newId);
    }
}

Здесь определен метод setIntegerId(), який отримує об'єкт Person и новое значение id для установки. Такой метод сможет работать с типуми Person<Integer>, Person<Number> и Person<Object>.

Однак при чтении (наприклад, при вызове getId()) мы можемо быть уверены лише в том, что получим Object, оскільки неизвестно, якой именно супертип використовується (Integer, Number або Object):


public class Program{
       
    public static void main(String[] args) {
           
        Person<Integer> pInt = new Person<>(1, "Tom");
        Person<Number> pNum = new Person<>(2.0, "Bob");
        Person<Object> pObj = new Person<>("F-456", "Sam");

        printPersonId(pInt);
        printPersonId(pNum);
        printPersonId(pObj);
    }

    static void printPersonId(Person<? super Integer> person) {

        // Здесь id - це Object
        Object id = person.getId();

        // Integer id = person.getId();  // так ми не можемо написать

        System.out.println(id); 
    }
}

Принцип PECS Producer extends

Принцип PECS (Producer Extends, Consumer Super) представляє простое правило помогает запомнить, когда якой wildcard використовувати:

  • постачає: об'єкт Consumer super дані (операции чтения, як в getId()). В цем случае случае використовується ? extends Type

  • споживає: об'єкт Назад дані (операции записи, як в setId()). В цем случае случае використовується ? super Type
