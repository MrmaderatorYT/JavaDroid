Синхронізація потіків. Оператор synchronized

Останнє оновлення: 22.10.2025

Состояние гонки

При работе потіки нередко обращаются к якимось общим ресурсам, которые визначенийы вне потіка, наприклад, обращение к якой-то змінної або даже файлу. Если одновременно декілька потіков обратятся к общему ресурсу, то результати виконання програми можуть быть неожиданными и даже непредсказуемыми. Наприклад, визначимо следующий код:


public class Program {

    public static void main(String[] args) {
        
        CommonResource commonResource= new CommonResource();

        for (int i = 1; i < 6; i++){
            
            new Thread(new CountThread(commonResource), "Thread "+ i).start();
        }
    }
}

class CommonResource{
    
    int x = 0;
}

class CountThread implements Runnable{

    CommonResource res;

    CountThread(CommonResource res){
        this.res = res;
    }

    public void run(){
		res.x = 1;

        for (int i = 1; i < 5; i++){

			System.out.printf("%s %d \n", Thread.currentThread().getName(), res.x);
			res.x++;

            try{
				Thread.sleep(100);
            }
            catch(InterruptedException _){}
        }
    }
}

Здесь визначений клас CommonResource, який представляє общий ресурс и в котором визначенийо одно целочисленное поле x.

Цет ресурс використовується класом потіка CountThread. Цет клас просто увеличивает в цикле значення x на единицу. Причем при входе в потік значення x=1:

res.x=1;

То есть в итоге мы ожидаем, что після виконання цикла res.x буде равно 4.

В главном класе програми запускуєся пять потіков. То есть мы ожидаем, что каждый потік буде увеличивать значення res.x с 1 до 4 и так пять раз. Но якщо мы посмотрим на результат работы програми, то он буде иным:

То есть поки один потік не окончил работу с полем res.x, с ним начинает працювати другой потік. Эта ситуация називається состоянием гонки (race condition), при котором два и болїї потіков борятся за управление одинм и тем же ресурсом. И якщо наше додаток виконує якие-то критически важные операції, наприклад, це финансовое додаток, где декілька потіков одновременно обращаются к счету пользователя для изменения баланса средств, то підобное состояние гонки может существенно нарушить корректность данных.

Щоб избежать підобной ситуации, треба синхронизировать потіки. Одним из способов синхронизации є використання ключевого слова synchronized. Цет оператор предваряет блок коду або метод, який підлежит синхронизации.

synchronized-блоки

synchronized-блок або синхронизированный блок представляє блок коду, к которому одномоментно имїїт доступ лише один потік. Синхронизированный блок имїїт следующую форму:


synchronized (obj)
{
    // критическая секция
}

При створенні синхронизированного блока коду після оператора synchronized идет об'єкт-заглушка: synchronized(obj). Причем в качестве об'єкта может використовуватися лише об'єкт якого-нибудь класа, но не примитивного типу. Затем идет блок коду, в яку помещается критическая секция - код, к которому в одним момент времени имїїт доступ лише один потік.

Каждый об'єкт в Java имїїт ассоциированный с ним монитор. Монитор представляє своего рода инструмент для управления доступа к об'єкту. Когда виконання коду доходит до оператора synchronized, монитор об'єкта блокуєся, и на час его блокировки монопольный доступ к блоку коду имїїт лише один потік, який и произвел блокировку. Після окончания работы блока коду, монитор об'єкта звільняєся и становится доступным для других потіков.

Після освобождения монитора его захоплює другой потік, а все остальные потіки продолжают ожидать его освобождения.

Наприклад изменим предудущий пример, пріменів синхронизированные блоки для синхронизации потіков:


public class Program {
 
    public static void main(String[] args) {
         
        CommonResource commonResource= new CommonResource();
        for (int i = 1; i < 6; i++){
             
            new Thread(new CountThread(commonResource), "Thread "+ i).start();
        }
    }
}
 
class CommonResource{
     
    int x=0;
}
 
class CountThread implements Runnable{
 
    CommonResource res;

    CountThread(CommonResource res){ this.res = res; }

    public void run(){

        synchronized(res){  // определяем синхронизированный блок

            // до конца блок идет критическая секция
            res.x = 1;

            for (int i = 1; i < 5; i++){
                System.out.printf("%s %d \n", Thread.currentThread().getName(), res.x);

                res.x++;

                try{
                    Thread.sleep(100);
                }
                catch(InterruptedException _){}
            }
        }
    }
}

И так, у нас есть неякий участок коду, к которому треба дать доступ одномоментно лише одному потіку. В примере выше це код, связанный с змінам общего ресурса - res. Тому эту часть коду оформляем в синхронизированный блок:


synchronized(res){  // определяем синхронизированный блок

    res.x = 1;

    for (int i = 1; i < 5; i++){
        System.out.printf("%s %d \n", Thread.currentThread().getName(), res.x);

        res.x++;

        try{
            Thread.sleep(100);
        }
        catch(InterruptedException _){}
        }
    }
}

Первый из потіков, який начнет выполнять цей код, захватит об'єкт res и начнет выполнять код synchronized-блока. Все остальные потіки в це час будуть ждать освобождения ресурса res. Когда первый потік завершит виконання коду в блоке, ресурс res звільняєся, и его захоплює другой (произвольный) потік. И опять остальные потіки будуть ожидать его освобождения. И так далїї, поки все потіки не выполнят код synchronized-блока. В итоге консольный вивід изменится и станет упорядоченным:

Синхронизированные методи

При применении оператора synchronized к методу, як и в случае с блоком, монопольный доступ к методу (выполнять метод в один момент времени) может лише один потік - первый, який начал его виконання. Для применения оператора synchronized к методу, изменим класы програми:


public class Program {
 
    public static void main(String[] args) {
         
        CommonResource commonResource= new CommonResource();
        for (int i = 1; i < 6; i++){
             
            new Thread(new CountThread(commonResource), "Thread "+ i).start();
        }
    }
}
 
class CommonResource{
     
    int x;
    synchronized void increment(){  // синхронизированный метод

        x = 1;
        for (int i = 1; i < 5; i++){
            System.out.printf("%s %d \n", Thread.currentThread().getName(), x);
            x++;
            try{
                Thread.sleep(100);
            }
            catch(InterruptedException _){}
        }
    }
}
 
class CountThread implements Runnable{
 
    CommonResource res;
    CountThread(CommonResource res){

        this.res = res;
    }
     
    public void run(){
        res.increment();
    }
}

Результат работы у даному випадку буде аналогичен примеру выше с блоком synchronized. Здесь опять в дело вступает монитор об'єкта CommonResource - общего об'єкта для всех потіков. Тому синхронизированным оголошується не метод run() в класе CountThread, а метод increment класа CommonResource. Когда первый потік начинает виконання методу increment, он захоплює монитор об'єкта CommonResource. А все потіки також продолжают ожидать его освобождения.
