Файли JAR, їх створення та виконання

Останнє оновлення: 21.09.2025

При компиляции программа на мові Java компилируется в набор классов, которые определены в программе и которые повинні располагаться в определенных каталогов в соответствии с их пакетами. Рднако це усложняет распространение программы. И чтобы программу на Java буво проще распространять, буви разработаны файлы JAR (Java ARchive або АрхивыJava). При цем JAR-файл может містятьь як файлы классов, так и другие типы файлов, наприклад, изображения и звуковые файлы, которые можуть застосовуватися в программе.

По сути JAR-файл - це способ об'єднати множество файлов в один, что упрощает их распространение и использование. JAR-файлы часто використовуються для упаковки библиотек, приложений и плагинов.

JAR-файл использует формат ZIP для организации файлов и подкаталогов и містить декілька файлов классов и подкаталогов в сжатом формате, что экономит место и повышает производительность. То есть по сути JAR-файл - це zip-архив, и для просмотра его содержимого можна використовувати любую утилиту ZIP. Если нам необхідно подключить в проект якую-то внешнюю библиотеку с потрібной нам функциональностью, то обычно библиотека надаєся в виде JAR-файла.

Для работы с JAR-файлами застосовується утилита jar, которая устанавливается в рамках JDK и располагается в каталоге jdk/bin. В общем случае эта команда имїїт следующий формат:


jar options file1 file2 ... fileN

Команде jar сначала передаются опции (options), которые определяют выполняемые действия. А затем через пробел передаются файлы, над которыми треба произвести действия. Список применяемых параметров (они аналогичны параметрам команды tar в UNIX):

  • c: створює новый або пустой архив и додає в него файлы. Если якие-або из указанных имен файлов є каталогами, программа jar обробляє их рекурсивно.

  • C: временно змінює каталог. Наприклад, следующая команда змінює подкаталог classdir для добавления файлов классов

    
jar cvf jarFileName.jar -C classdir *.class

  • e: створює точку входа в манифесте.

  • f: устанавливает имя JAR-файла, которое передаєся в качестве второго аргумента командного рядка.

  • i: створює индексный файл (для ускорения поиска в большом архиве).

  • m: додає манифест в JAR-файл. У каждого архива есть манифест по умолчанию, но можна установить свой собственный.

  • M: указывает, что не треба создаваит файл манифеста.

  • t: отображает оглавление.

  • u: обновляет существующий JAR-файл.

  • v: формирует подробный вывод.

  • x: извлекает файлы. Если указано одно або декілька имен файлов, извлекаются лише эти файлы. В противном случае извлекаются все файлы.

  • 0: сохраняет без сжатия ZIP.

Створення JAR-файлов

Наиболїї распространенная команда для создания нового JAR-файла использует следующий синтаксис:


jar cvf jarFileName file1 file2 ... fileN

Наприклад, пусть у нас есть файл Program.java с простеньким кодом:


class Program{

    public static void main(String[] args) {
        
       System.out.println("Hello ");
    }
}

Сначала скомпилируем файл за допомогою компилятора javac:


javac Program.java

В итоге компилируется файл Program.class. Затем создадим из цього класса jar-файл:


jar cvf app.jar Program.class

В данном случае опция "cvf" фактически представляє об'єднание трех опций - "c", "v", и "f". В качестве имени выходного файла устанавливается "app.jar". Полный вывод команды:


eugene@Eugene:/workspace/java$ jar cvf app.jar Program.class
added manifest
adding: Program.class(in = 425) (out= 292)(deflated 31%)
eugene@Eugene:/workspace/java$

И после виконання команды в текущей папке появится файл "app.jar".

Стои відзначити, что мы можемо передать конкретные файлы классов на створення архива:


jar cvf app.jar Program.class Person.class Employee.class

Но якщо нам треба указать все класса, которые находятся в определенной папке, то мы можемо використовувати символ подстановки "*":


jar cvf app.jar *.class

В данном случае в jar-архив добавляются все классы, которые находятся в текущей папке. Если классы располагаются в якой-то определенной папке, наприклад, в папке "com/", то треба указать полный путь к классам:


jar cvf app.jar com//Program.class com//Person.class com//Employee.class

// або так 
jar cvf app.jar com//*.class

Содержимое jar-файла

Распакуем цей файл за допомогою любой программы, которая разархивирует файлы zip. Наприклад, на Linux обычно в системе установлена утилита unzip:


eugene@Eugene:/workspace/java$ unzip -l app.jar
Archive:  app.jar
  Length      Date    Time    Name
---------  ---------- -----   ----
        0  2025-09-19 14:23   META-INF/
       62  2025-09-19 14:23   META-INF/MANIFEST.MF
      425  2025-09-19 14:21   Program.class
---------                     -------
      487                     3 files
eugene@Eugene:/workspace/java$

С помощью опции -l команде unzip указывается, что треба вывести содержимое архива на консоль. И в данном случае мы видим, что архив вже имїїт деяку структуру:

  • Папка META-INF

    • Файл MANIFEST.MF: файл манифеста

  • Файл Program.class: файл основного класса программы

Таким чином, якщо мы распакуем jar-файл, то мы увидим, что он також містить файл манифеста.

Запуск jar-файлов

Java дозволяє напрямую запускать jar-файл як обычную программу. Для цього утилите java передаєся параметр -jar и путь к jar-архиву. Наприклад, запустим выше створений файл app.jar:


eugene@Eugene:/workspace/java$ java -jar app.jar
no main manifest attribute, in app.jar
eugene@Eugene:/workspace/java$

И мы видим, что java не запустила файл, оскільки в манифестве jar-архива отсутствует атрибут "Main-Class", о чем собственно говорит отображенное на консоли сообщение.

Створення исполняемого файла jar

Однак, чтобы сделать jar-файл исполняемым, нам необязательно создавать або изменять манифест, для цього мы можемо воспользоваться опцией "e" команды jar. В частности, выполним следующую команду:


jar cvfe app.jar Program Program.class

После названия jar-файла команде передаєся имя класса без расширения ".class", то есть в нашем случае Program. И таким чином, мы получим исполняемый jar, який мы сможемо запустить за допомогою команды java -jar app.jar:


eugene@Eugene:/workspace/java$ java -jar app.jar
Hello 
eugene@Eugene:/workspace/java$

Варто відзначити, что имя класса указывается с учетом всех пакетів, в которых расположен данный класс. Наприклад, якщо бы класс Program располагался бы в пакеті "com." (то есть полное имя класса буво бы "com..Program"), то в качестве главного класса программы указывался бы com..Program


jar cvfe app.jar com..Program com//Program.class

В зависимости от конфигурации операционной системы пользователи можуть даже запускать приложение, дважды щелкнув значок JAR-файла.

Наприклад, в Windows установщик Java створює ассоциацию файла с расширением ".jar", которая запускає файл за допомогою команды javaw -jar. В отличие от команды java, команда javaw не открывает окно оболочки.

В Mac OS распознается расширение файла «.jar», и при двойном щелчке по JAR-файлу запускаєся программа Java, которая, в свою очередь, запускає jar-файл.

Манифест jar-файла

Манифест представляє описание содержимого архива. У каждого архива есть свой манифест. Можна предоставить свой файл манифеста. Если мы его не предоставим, манифест створюєся по умолчанию.

Файл манифеста називається MANIFEST.MF и находится в JAR-файле в специальном подкаталоге META-INF. По факту це текстовый файл, який в самом минимальном виде містить версию манифеста в виде параметра Manifest-Version:

Manifest-Version: 1.0

Но також можуть містятьь и большїї количество записей. Наприклад, при компиляции в создаваемый манифест також обычно додаєся версия использованного JDK:

Manifest-Version: 1.0
Created-By: 25 (Oracle Corporation)

Записи манифеста группируются по разделам. Первый раздел називається основным - он застосовується ко всему файлу. В целом даже для основного раздела имется большїї количество атрибутов. Все их можна посмотреть в документации. Отмечу основные из них:

  • Manifest-Version: Визначає версию файла манифеста. Значение — допустимый номер версии.

  • Created-By: Визначає версию и вендора реализации Java, на основе которой сгенерирован цей файл манифеста. Цет атрибут генеруєся инструментом jar.

  • Signature-Version: Визначає версию подписи JAR-файла. Значение повинно быть допустимой строкой с номером версии.

  • Class-Path: Значение цього атрибута указывает относительные URL-адреса библиотек, необходимых цему приложению. URL-адреса разделяются одним або несколькими пробелами. Загрузчик классов приложения использует значение цього атрибута для построения своего внутреннего пути поиска. Подробнїї см. в разделе «Атрибут Class-Path».

  • Multi-Release: Цет атрибут определяет, є ли цей JAR-файл многорелизным. Если значение равно "true", регистр игнорируется, JAR-файл буде обработан средой виконання Java и инструментами як многорелизный JAR-файл. В противном случае, якщо значение отлично от "true", цей атрибут игнорируется.

  • Main-Class: Значение цього атрибута — имя основного класса приложения, який модуль запуска буде загружать при запуске. Имя класса указывается без расширения .class.

Наприклад, атрибут Main-Class застосовується для указания главного класса, який повинен використовуватися для запуска программы. Если мы хочемо, чтобы jar-архив був исполняемым, то один из способов решения проблемы - указать цей атрибут в файле манифеста.

Створення и добавление манифеста

Допустим, у нас есть простая программка в файле Program.java:


class Program{

    public static void main(String[] args) {
        
       System.out.println("Hello ");
    }
}

Скомпилируем эту программу:


javac Program.java

В итоге компилируется файл Program.class.

Затем создадим в той же папке текстовый файл manifest.mf, який буде служить файлом манифеста и який буде иметь следующїї содержимое:

Manifest-Version: 1.0
Main-Class: Program

Здесь атрибуту "Main-Class" передаем название класса - Program без расширения .class. Варто відзначити, что имя класса указывается с учетом всех пакетів, в которых расположен данный класс. Наприклад, якщо бы класс Program располагался бы в пакеті "com." (то есть полное имя класса буво бы "com..Program"), то указывался бы атрибут Main-Class: com..Program

Далїї создадим из класса Program.class и манифеста manifest.mf jar-файл:


jar cfm app.jar manifest.mf Program.class

В итоге мы получим файл app.jar. И поскольку мы указали в манифесте главный файл программы, мы можемо запустить jar-файл на выполнение за допомогою следующей команды java -jar app.jar:


eugene@Eugene:/workspace/java$ java -jar app.jar
Hello 
eugene@Eugene:/workspace/java$

Обновление манифеста

Також отмечу, что мы можемо обновить файл манифеста. Щоб обновить манифест существующего JAR-файла, достаточно поместить дополнения в текстовый файл и використовувати следующую команду:


jar ufm app.jar manifest.mf
