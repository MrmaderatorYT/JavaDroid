Файли JAR, їх створення та виконання

Останнє оновлення: 21.09.2025

При компиляции програма на мові Java компилируется в набор класов, которые визначенийы в програме и которые повинні располагаться в визначенийных каталогов в соответствии с их пакетуми. Рднако це усложняет распространение програми. И чтобы програму на Java буво проще распространять, буви разработаны файли JAR (Java ARchive або АрхивыJava). При цем JAR-файл может містятьь як файли класов, так и другие типи файлів, наприклад, изображения и звуковые файли, которые можуть застосовуватися в програме.

По сути JAR-файл - це способ об'єднати множина файлів в один, что упрощает их распространение и використання. JAR-файли часто використовуються для упаковки библиотек, приложений и плагинов.

JAR-файл використовує формат ZIP для организации файлів и підкаталогов и містить декілька файлів класов и підкаталогов в сжатом формате, что экономит место и повышает производительность. То есть по сути JAR-файл - це zip-архив, и для просмотра его содержимого можна використовувати любую утилиту ZIP. Если нам необхідно підключить в проєкт якую-то внешнюю бібліотеку с потрібной нам функциональностью, то обычно бібліотека надаєся в виде JAR-файлу.

Для работы с JAR-файлуми застосовується утилита jar, которая встановлюєся в рамках JDK и располагается в каталогу jdk/bin. В общем случае эта команда имїїт следующий формат:


jar options file1 file2 ... fileN

Команде jar сначала передаються опции (options), которые визначають выполняемые действия. А затем через пробел передаються файли, над которыми треба произвести действия. Список застосовуємоых параметрів (они аналогичны параметрам команди tar в UNIX):

  • c: створює новый або порожній архив и додає в него файли. Если якие-або из указанных имен файлів є каталогуми, програма jar обробляє их рекурсивно.

  • C: временно змінює каталог. Наприклад, следующая команда змінює підкаталог classdir для добавления файлів класов

    
jar cvf jarFileName.jar -C classdir *.class

  • e: створює точку входа в манифесте.

  • f: встановлює ім'я JAR-файлу, которое передаєся в качестве второго аргумента командного рядка.

  • i: створює индексный файл (для ускорения поиска в большом архиве).

  • m: додає манифест в JAR-файл. У каждого архива есть манифест по умолчанию, но можна установить свой собственный.

  • M: вказує, что не треба создаваит файл манифеста.

  • t: отображает оглавление.

  • u: обновляет существующий JAR-файл.

  • v: формирует підробный вивід.

  • x: извлекает файли. Если указано одно або декілька имен файлів, извлекаются лише эти файли. В противном случае извлекаются все файли.

  • 0: сохраняет без сжатия ZIP.

Створення JAR-файлів

Наиболїї распространенная команда для створенняя нового JAR-файлу використовує следующий синтаксис:


jar cvf jarFileName file1 file2 ... fileN

Наприклад, пусть у нас есть файл Program.java с простеньким кодом:


class Program{

    public static void main(String[] args) {
        
       System.out.println("Hello ");
    }
}

Сначала скомпилируем файл за допомогою компилятора javac:


javac Program.java

В итоге компилируется файл Program.class. Затем створимо из цього класа jar-файл:


jar cvf app.jar Program.class

У даному випадку опция "cvf" фактически представляє об'єднание трех опций - "c", "v", и "f". В качестве імені выходного файлу встановлюєся "app.jar". Полный вивід команди:


eugene@Eugene:/workspace/java$ jar cvf app.jar Program.class
added manifest
adding: Program.class(in = 425) (out= 292)(deflated 31%)
eugene@Eugene:/workspace/java$

И після виконання команди в текущей папке появится файл "app.jar".

Стои відзначити, что мы можемо передать конкретные файли класов на створення архива:


jar cvf app.jar Program.class Person.class Employee.class

Но якщо нам треба указать все класа, которые находятся в визначенийной папке, то мы можемо використовувати символ підстановки "*":


jar cvf app.jar *.class

У даному випадку в jar-архив додаються все класы, которые находятся в текущей папке. Если класы располагаются в якой-то визначенийной папке, наприклад, в папке "com/", то треба указать полный путь к класам:


jar cvf app.jar com//Program.class com//Person.class com//Employee.class

// або так 
jar cvf app.jar com//*.class

Вміст jar-файлу

Распакуем цей файл за допомогою любой програми, которая разархивирует файли zip. Наприклад, на Linux обычно в системе установлена утилита unzip:


eugene@Eugene:/workspace/java$ unzip -l app.jar
Archive:  app.jar
  Length      Date    Time    Name
---------  ---------- -----   ----
        0  2025-09-19 14:23   META-INF/
       62  2025-09-19 14:23   META-INF/MANIFEST.MF
      425  2025-09-19 14:21   Program.class
---------                     -------
      487                     3 files
eugene@Eugene:/workspace/java$

За допомогою опции -l команде unzip вказуєся, что треба вывести вміст архива на консоль. И у даному випадку мы видим, что архив вже имїїт деяку структуру:

  • Папка META-INF

    • Файл MANIFEST.MF: файл манифеста

  • Файл Program.class: файл основного класа програми

Таким чином, якщо мы распакуем jar-файл, то мы увидим, что он також містить файл манифеста.

Запуск jar-файлів

Java дозволяє напрямую запускуть jar-файл як обычную програму. Для цього утилите java передаєся параметр -jar и путь к jar-архиву. Наприклад, запустимо выше створений файл app.jar:


eugene@Eugene:/workspace/java$ java -jar app.jar
no main manifest attribute, in app.jar
eugene@Eugene:/workspace/java$

И мы видим, что java не запустила файл, оскільки в манифестве jar-архива отсутствует атрибут "Main-Class", о чем собственно говорит отображенное на консоли сообщение.

Створення исполняемого файлу jar

Однак, чтобы сделать jar-файл исполняемым, нам необязательно создавать або изменять манифест, для цього мы можемо воспользоваться опцией "e" команди jar. Зокрема, виконаємо следующую команду:


jar cvfe app.jar Program Program.class

Після названия jar-файлу команде передаєся ім'я класа без расширения ".class", то есть в нашем случае Program. И таким чином, мы получим исполняемый jar, який мы сможемо запустить за допомогою команди java -jar app.jar:


eugene@Eugene:/workspace/java$ java -jar app.jar
Hello 
eugene@Eugene:/workspace/java$

Варто відзначити, что ім'я класа вказуєся с учетом всех пакетів, в которых расположен данный клас. Наприклад, якщо бы клас Program располагался бы в пакеті "com." (то есть полное ім'я класа буво бы "com..Program"), то в качестве главного класа програми указывался бы com..Program


jar cvfe app.jar com..Program com//Program.class

В зависимости от конфигурации операционной системы пользователи можуть даже запускуть додаток, дважды щелкнув значок JAR-файлу.

Наприклад, в Windows установщик Java створює ассоциацию файлу с расширением ".jar", которая запускує файл за допомогою команди javaw -jar. В отличие от команди java, команда javaw не открывает окно оболочки.

В Mac OS распознается расширение файлу «.jar», и при двойном щелчке по JAR-файлу запускуєся програма Java, которая, в свою очередь, запускує jar-файл.

Манифест jar-файлу

Манифест представляє описание содержимого архива. У каждого архива есть свой манифест. Можна предоставить свой файл манифеста. Если мы его не предоставим, манифест створюєся по умолчанию.

Файл манифеста називається MANIFEST.MF и находится в JAR-файле в специальном підкаталогу META-INF. По факту це текстовый файл, який в самом минимальном виде містить версию манифеста в виде параметра Manifest-Version:

Manifest-Version: 1.0

Но також можуть містятьь и большїї кількість записей. Наприклад, при компиляции в создаваемый манифест також обычно додаєся версия использованного JDK:

Manifest-Version: 1.0
Created-By: 25 (Oracle Corporation)

Записи маніфесту групуються за розділами. Перший розділ називається основним - він застосовується до всього файлу. В цілому навіть для основного розділу наявна велика кількість атрибутів. Все их можна посмотреть в документации. Отмечу основные из них:

  • Manifest-Version: Визначає версию файлу манифеста. Значення — допустимый номер версии.

  • Created-By: Визначає версию и вендора реалізації Java, на основе которой сгенерирован цей файл манифеста. Цет атрибут генеруєся инструментом jar.

  • Signature-Version: Визначає версию підписи JAR-файлу. Значення повинно быть допустимой рядківой с номером версии.

  • Class-Path: Значення цього атрибута вказує относительные URL-адреса библиотек, необходимых цему приложению. URL-адреса розділяются одним або несколькими пробелами. Загрузчик класов додатки використовує значення цього атрибута для построения своего внутреннего пути поиска. Підробнїї см. в розділе «Атрибут Class-Path».

  • Multi-Release: Цет атрибут визначає, є ли цей JAR-файл многорелизным. Если значення равно "true", регистр игнорируется, JAR-файл буде обработан средой виконання Java и инструментами як многорелизный JAR-файл. В противном случае, якщо значення отлично от "true", цей атрибут игнорируется.

  • Main-Class: Значення цього атрибута — ім'я основного класа додатки, який модуль запуску буде загружать при запуску. Имя класа вказуєся без расширения .class.

Наприклад, атрибут Main-Class застосовується для указания главного класа, який повинен використовуватися для запуску програми. Если мы хочемо, чтобы jar-архив був исполняемым, то один из способов решения проблемы - указать цей атрибут в файле манифеста.

Створення и додавання манифеста

Допустим, у нас есть простая програмка в файле Program.java:


class Program{

    public static void main(String[] args) {
        
       System.out.println("Hello ");
    }
}

Скомпилируем эту програму:


javac Program.java

В итоге компилируется файл Program.class.

Затем створимо в той же папке текстовый файл manifest.mf, який буде служить файлом манифеста и який буде иметь следующїї вміст:

Manifest-Version: 1.0
Main-Class: Program

Здесь атрибуту "Main-Class" передаємо название класа - Program без расширения .class. Варто відзначити, что ім'я класа вказуєся с учетом всех пакетів, в которых расположен данный клас. Наприклад, якщо бы клас Program располагался бы в пакеті "com." (то есть полное ім'я класа буво бы "com..Program"), то указывался бы атрибут Main-Class: com..Program

Далїї створимо из класа Program.class и манифеста manifest.mf jar-файл:


jar cfm app.jar manifest.mf Program.class

В итоге мы получим файл app.jar. И поскольку мы указали в манифесте главный файл програми, мы можемо запустить jar-файл на виконання за допомогою следующей команди java -jar app.jar:


eugene@Eugene:/workspace/java$ java -jar app.jar
Hello 
eugene@Eugene:/workspace/java$

Обновление манифеста

Також отмечу, что мы можемо обновить файл манифеста. Щоб обновить манифест существующего JAR-файлу, достатньо поместить дополнения в текстовый файл и використовувати следующую команду:


jar ufm app.jar manifest.mf
