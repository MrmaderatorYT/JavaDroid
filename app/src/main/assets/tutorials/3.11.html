Узагальнення (Generics)

Останнє оновлення: 08.10.2025

Узагальнення або generics (обобщенные типы и методы) дозволяють нам уйти от жесткого определения используемых типов и визначати классы/інтерфейсы и методы, которые не привязаны к определенным типум. Благодаря обобщениям возможна написание кода, який можна повторно використовувати для различных типов.

Щоб розібратися особливо цього явища, спочатку подивимося на проблему, що могла виникнути до появи узагальнених типів. Подивимося на прикладі. Допустимо, ми визначаємо клас для зберігання даних користувача:


class Person {

    private int id;
    private String name;

    int getId(){ return id; }
    String getName(){ return name; }

    Person(int id, String name)
    {
        this.id = id; 
        this.name = name;
    }
}

Клас Person визначає два поля: id – унікальний ідентифікатор користувача та name – ім'я користувача. Причому тут ідентифікатор користувача заданий як числове значення, тобто це значення 1, 2, 3, 4 і так далі.

Однак нерідко для ідентифікатора використовуються і рядкові значення. І у числових, і у рядкових значень є свої плюси та мінуси. І на момент написання класу ми можемоо точно не знати, що краще вибрати для зберігання ідентифікатора – рядки чи числа. Або, можливо, цей клас буде використовуватись іншими розробниками, які можуть мати свою думку щодо цієї проблеми, наприклад, вони можуть для подання ідентифікатора створити спеціальний клас.

И на первый взгляд, чтобы выйти из подобной ситуации, можна определить поле id як поле типу Object. Так як тип Object є универсальным типом, от которого найдеся все типы, соответственно в подобном поле мы можемо сохранить и строки, и числа:


class Person {

    private Object id;      // универсальный идентификатор - и числа, и строки
    private String name;

    Object getId(){ return id; }
    String getName(){ return name; }

    Person(Object id, String name)
    {
        this.id = id; 
        this.name = name;
    }
}

Далі у програмі ми можемоо використовувати цей клас, передаючи в його конструктор для id та числа, і рядки:


class Program {

    public static void main(String[] args){
        
        var tom = new Person(546, "Tom");
        tom.print();	// Person. Id: 546;  Name: Tom

        var bob = new Person("zpio9", "Bob");
        bob.print();	// Person. Id: zpio9;  Name: Bob
    } 
}

class Person {

    private Object id;      // универсальный идентификатор - и числа, и строки
    private String name;

    Object getId(){ return id; }
    String getName(){ return name; }

    Person(Object id, String name)
    {
        this.id = id; 
        this.name = name;
    }
}

Все ніби чудово працює, але таке рішення є не двже оптимальним. Справа в тому, що в даному випадку ми стикаємося із перетвореннями.

Так, при передаче в конструктор значения типу int, происходит упаковка цього значения в тип Integer с понаступним преобразованием в тип Object (int -> Integer -> Object):


var tom = new Person(546, "Tom");	// преобразование значения int в тип Integer и Object

Що стосується рядком також є висхідне перетворення (String -> Object), але менш критично:

var bob = new Person("zpio9", "Bob");

Але якщо висхідне перетворення відбувається автоматично, то щоб отримати дані в змінну типів int або String, нам необхідно явним чином виконати перетворення типів (низхідне перетворення):


int tomId = (int)tom.getId();		// Object -> int
String bobId = (String) bob.getId();		// Object -> String

Інша проблема, з якою ми тут можемоо зіткнутися, – це проблема безпеки типів. Так, ми отримаємо помилку під час виконання програми, якщо напишемо так:


class Program {

    public static void main(String[] args) {
        
        var tom = new Person(546, "Tom");
        String tomId = (String)tom.getId();  // !Ошибка  - Исключение java.lang.ClassCastException
        System.out.printf("tom id: %s\n", tomId);
    } 
}

Мы можемо не знать, якой именно об'єкт представляє id, и при попытке отримати число в данном случае мы столкнемся с исключением java.lang.ClassCastException. Причем с исключением мы столкнемся на этапе виконання програми, то есть на этапе компиляции мы нияк не сможемо отследить эту проблему.

Проблема может покизаться искуственной, оскільки в данном случае мы видим, что в конструктор передаєся число, поцему мы вряд ли будем пытаться преобразовывать его в строку. Однак в процессе разработки мы можемо не знать, якой именно тип представляє значение в id, и при попытке отримати значение в потрібном нам типе мы можемо столкнуться с исключением java.lang.ClassCastException.

Писати для кожного окремого типу свою версію класу Person теж не є добрим рішенням, тому що в цьому випадку ми змушені повторюватися.

Для решения цих проблем в язык Java бува добавлена поддержка узагальнень - обобщенных типов и методів, которые також часто называют универсальными типуми и методами, а по английский generics. Обобщения дозволяють указать конкретный тип, який буде використовуватися. Тому определим класс Person як обощенный:


class Person<T> {

    private T id;  
    private String name;

    T getId(){ return id; }
    String getName(){ return name; }

    Person(T id, String name)
    {
        this.id = id; 
        this.name = name;
    }
}

Угловые скобки в описании class Person<T> указывают, что класс є обобщенным, а тип T, заключенный в угловые скобки, буде використовуватися цим классом. Необязательно використовувати именно букву T, це может быть и любая другая буква або набор символов. Причем сейчас на этапе написания кода нам неизвестно, что це буде за тип, це может быть любой тип. Тому параметр T в угловых скобках ще називається універсальним параметром, оскільки вместо него можна подставить любой тип.

Метод getId() повертає значение змінної id, но оскільки данная переменная представляє тип T, то данный метод також повертає об'єкт типу T: T getId().

Наприклад, замість параметра T можна використовувати об'єкт Integer, тобто число, яке представляє номер користувача. Це також може бути об'єкт String, або будь-який інший тип (клас або навіть інтерфейс):


class Program {

    public static void main(String[] args){
        
        Person<Integer> tom = new Person<Integer>(546, "Tom");   // преобразование типов не потрібно
        Person<String> bob = new Person<String>("abc123", "Bob"); // преобразование типов не потрібно
        
        Integer tomId = tom.getId();      // преобразование типов не потрібно
        String bobId = bob.getId();  // преобразование типов не потрібно
        
        System.out.println(tomId);   // 546
        System.out.println(bobId);   // abc123
    } 
}

class Person<T> {

    private T id;  
    private String name;

    T getId(){ return id; }
    String getName(){ return name; }

    Person(T id, String name)
    {
        this.id = id; 
        this.name = name;
    }
}

При определении змінної даннного класса и создании об'єкта после имени класса в угловых скобках потрібно указать, якой именно тип буде використовуватися вместо универсального параметра. Наприклад, первый об'єкт буде використовувати тип Integer, то есть вместо T буде подставляться Integer:

Person<Integer> tom = new Person<Integer>(546, "Tom");

І в даному випадку фактично ми працюватимемо з класом


class Person {

    private Integer id;  
    private String name;

    Integer getId(){ return id; }
    String getName(){ return name; }

    Person(Integer id, String name)
    {
        this.id = id; 
        this.name = name;
    }
}

При цем треба учитывать, что они работают лише с об'єктами, но не работают с примитивными типуми. То есть мы можемо написать Person<Integer>, но не можемо використовувати тип int або double, наприклад, Person<int>. Вместо примитивных типов треба використовувати классы-обертки: Integer вместо int, Double вместо double и т.д.

А второй об'єкт использует тип String:

Person<String> bob = new Person<String>("abc123", "Bob");

При спробі передати для параметра id значення іншого типу отримаємо помилку компіляції:


Person<Integer> tom = new Person<Integer>("546", "Tom");  // ошибка компиляции

А при получении значения через getId() нам больше не потребуется операция приведения типов:

Integer tomId = tom.getId();      // преобразование типов не потрібно

Тим самим ми уникнемо проблем з типобезпекою та необхідності виконувати перетворення типів. Якщо ж виникне проблема з перетвореннями, то компілятор виявить ці проблеми вже на етапі компіляції. Відповідно ми не отримаємо програму, що падає. Таким чином, використовуючи узагальнений варіант класу, ми знижуємо час на виконання та кількість потенційних помилок.

Універсальний параметр як узагальнений тип

У цьому універсальний параметр може представляти узагальнений тип:


// класс компании
class Company<P>{

    private P ceo;  // президент компании
    P getCEO(){ return ceo; }

    Company(P ceo){

        this.ceo = ceo;
    }
}

class Person<T> {

    private T id;  
    private String name;

    T getId(){ return id; }
    String getName(){ return name; }

    Person(T id, String name)
    {
        this.id = id; 
        this.name = name;
    }
}

Тут клас компанії визначає поле CEO, яке зберігає президента компанії. І ми можемоо передати цього властивості значення типу Person, типизированного яким-небудь типом:


class Program {

    public static void main(String[] args){
        
        Person<Integer> tom = new Person<Integer>(546, "Tom");
        Company<Person<Integer>> melkosoft=  new Company<Person<Integer>>(tom);
        
        var ceo = melkosoft.getCEO();
        System.out.println(ceo.getId());  // 546
        System.out.println(ceo.getName());  // Tom
    } 
}

Використання кількох універсальних параметрів

Узагальнення можуть використовувати кілька універсальних параметрів одночасно, які можуть представляти однакві чи різні типи:



class Person<T, K> {

    private T id;  
    private K password;  
    private String name;

    T getId(){ return id; }
    K getPassword(){ return password; }
    String getName(){ return name; }

    Person(T id, String name, K pass)
    {
        this.id = id; 
        this.name = name;
        this.password = pass;
    }
}

Тут клас Person використовує два універсальні параметри: один параметр для ідентифікатора id, інший параметр – для поля, яке представляє пароль (оскільки пароль може існувати у різних формах – текстовий, голосовий тощо). Застосуємо цей клас:

class Program {

    public static void main(String[] args){
        
        Person<Integer, String> tom = new Person<Integer, String>(546, "Tom", "qwerty12345");
        System.out.println(tom.getId());  // 546
        System.out.println(tom.getPassword());  // qwerty12345
    } 
}

Здесь об'єкт Person типизируется типуми Integer и String. То есть в качестве универсального параметра T використовується тип Integer, а для параметра K - тип String.

Узагальнені інтерфейси

Інтерфейси, як і класи, можуть бути узагальненими. Наприклад:


public class Program{
     
	public static void main(String[] args) {
         
		Accountable<String> acc1 = new Account("1235rwr", 5000);
		Account acc2 = new Account("2373", 4300);
		System.out.println(acc1.getId());
		System.out.println(acc2.getId());
	}
}
interface Accountable<T>{
	T getId();
	int getSum();
	void setSum(int sum);
}
class Account implements Accountable<String>{
	
    private String id;
	private int sum;
	
	Account(String id, int sum){
		this.id = id;
		this.sum = sum;
	}
	
    public String getId() { return id; }
	public int getSum() { return sum; }
	public void setSum(int sum) { this.sum = sum; }
}

У цьому випадку інтерфейс Accountable представляє базовий функціонал для банківського рахунку та оголошує методи повернення ідентифікатора та отримання/установки суми рахунку. А клас Account реалізує його.

При реалізації такого інтерфейсу є дві стратегії. В даному випадку реалізована перша стратегія, коли при реалізації для універсального параметра інтерфейсу задається конкретний тип, наприклад, у разі це тип String. Тоді клас, який реалізує інтерфейс, жорстко прив'язаний до цього типу.

Друга стратегія представляє визначення узагальненого класу, що також використовує той самий універсальний параметр:


public class Program{
     
	public static void main(String[] args) {
         
		Account<String> acc1 = new Account<String>("1235rwr", 5000);
		Account<String> acc2 = new Account<String>("2373", 4300);
		System.out.println(acc1.getId());
		System.out.println(acc2.getId());
	}
}
interface Accountable<T>{
	T getId();
	int getSum();
	void setSum(int sum);
}
class Account<T> implements Accountable<T>{
	
    private T id;
	private int sum;
	
	Account(T id, int sum){
		this.id = id;
		this.sum = sum;
	}
	
    public T getId() { return id; }
	public int getSum() { return sum; }
	public void setSum(int sum) { this.sum = sum; }
}

Узагальнені методи

Крім узагальнених типів можна також створювати узагальнені методи, які також використовуватимуть універсальні параметри. Наприклад:


public class Program{
     
	public static void main(String[] args) {
         
		Printer printer = new Printer();
		String[] people = {"Tom", "Alice", "Sam", "Kate", "Bob", "Helen"};
		Integer[] numbers = {23, 4, 5, 2, 13, 456, 4};
		printer.<String>print(people);
		printer.<Integer>print(numbers);
	}
}

class Printer{
	
	public <T> void print(T[] items){
		for(T item: items){
			System.out.println(item);
		}
	}
}

Особливістю узагальненого методу є використання універсального параметра в оголошенні методу після всіх модифікаторів та перед типом значення, що повертається.

public <T> void print(T[] items)

Потім усередині методу всі значення типу T будутьь представляти цей універсальний параметр.

При виклику подібного методу перед його ім'ям у кутових дужках вказується, який тип передаватиметься місце універсального параметра:


printer.<String>print(people);
printer.<Integer>print(numbers);

Однак в цем случае (як и в большинстве случаев) при вызове метода також можна опустить параметр типу (<String> / <Integer>). У компилятора достаточно информации, чтобы определить нужный вам метод. Он сопоставляет тип аргументов с обобщенным типом T и робить вывод, что T в первом случае повинен быть String и во втором - Integer. То есть, можна просто вызвать


Printer printer = new Printer();
String[] people = {"Tom", "Alice", "Sam", "Kate", "Bob", "Helen"};
Integer[] numbers = {23, 4, 5, 2, 13, 456, 4};
printer.print(people);
printer.print(numbers);

Узагальнені конструктори

Конструктори як і методи можуть бути узагальненими. У цьому випадку перед конструктором також вказуються у кутових дужках універсальні Параметри:


public class Program{
     
	public static void main(String[] args) {
         
		Person acc1 = new Person("cid2373", 5000);
		Person acc2 = new Person(53757, 4000);
		System.out.println(acc1.getId());
		System.out.println(acc2.getId());
	}
}

class Person{
	
    private String id;
	private int sum;
	
	<T>Person(T id, int sum){
		this.id = id.toString();
		this.sum = sum;
	}
	
    public String getId() { return id; }
	public int getSum() { return sum; }
	public void setSum(int sum) { this.sum = sum; }
}

У даному випадку конструктор приймає параметр id, який представляє тип T. У конструкторі його значення перетворюється на рядок і зберігається на локальну змінну.
