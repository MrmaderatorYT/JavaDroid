Інтерфейси у механізмі зворотного виклику

Останнє оновлення: 28.12.2015

Одним з найпоширеніших способів використання інтерфейсів Java є створення зворотного виклику. Суть зворотного виклику полягає в тому, що ми створюємо дії, що викликаються за інших дій. Тобто одні дії викликаються іншими діями. Стандартний приклад – натискання на кнопку. Коли ми натискаємо кнопку, ми робимо дію, але у відповідь це натискання запускаються інші дії. Наприклад, натискання піктограми принтера запускає друк документа на принтері та ін.

Розглянемо такий приклад:


public class EventsApp {

    public static void main(String[] args) {
        
        Button button = new Button(new ButtonClickHandler());
        button.click();
        button.click();
        button.click();
    }
}

class ButtonClickHandler implements EventHandler{
    
    public void execute(){
        
        System.out.println("Кнопка нажата!");
    }
}

interface EventHandler{
    
    void execute();
}

class Button{
    
    EventHandler handler;
    Button(EventHandler action){
        
        this.handler=action;
    }
    public void click(){
        
        handler.execute();
    }
}

Отже, тут у нас визначено клас Button, який у конструкторі приймає об'єкт інтерфейсу EventHandler та у методі click (імітація натискання) викликає Метод виконання цього об'єкта.

Далїї определяется реализация EventHandler в виде класса ButtonClickHandler. И в основной программе об'єкт цього класса передаєся в конструктор Button. Таким чином, через конструктор мы устанавливаем обработчик нажатия кнопки. И при каждом вызове метода button.click() буде вызываться цей обработчик.

У результаті програма виведе на консоль наступний результат:

Кнопка нажата!
Кнопка нажата!
Кнопка нажата!

Але здавалося б, навіщо нам виносити всі дії в інтерфейс, його реалізувати, чому б не написати простіше одразу у класі Button:


class Button{
    
    public void click(){
        
        System.out.println("Кнопка нажата!");
    }
}

Справа в тому, що на момент визначення класу нам не завжди бувають точно відомі ті дії, які мають проводитися. Особливо якщо клас Button та клас основної програми знаходяться у різних пакетах, бібліотеках та можуть проектуватися різними розробниками. До того ж, у нас може бути кілька кнопок - об'єктів Button і кожного об'єкта треба визначити свою дію. Наприклад, змінимо головний клас програми:


public class EventsApp {

    public static void main(String[] args) {
        
        Button tvButton = new Button(new EventHandler(){
              
            private boolean on = false;
            public void execute(){
                
                if(on) {
                    System.out.println("Телевизор выключен..");
                    on=false;
                }   
                else {
                    System.out.println("Телевизор включен!");
                    on=true;
                }
            }
        });
        
        Button printButton = new Button(new EventHandler(){
              
            public void execute(){
                
                System.out.println("Запущена печать на принтере...");
            }
        });
        
        tvButton.click();
        printButton.click();
        tvButton.click();
    }
}

Здесь у нас две кнопки - одна для включения-выключения телевизора, а другая для печати на принтере. Вместо того, чтобы создавать отдельные классы, реализующие інтерфейс EventHandler, здесь обработчики задаются в виде анонимных об'єктов, которые реализуют інтерфейс EventHandler. Причем обработчик кнопки телевизора хранит дополнительное состояние в виде логической змінної on.

У результаті консоль виведе наступний результат:

Телевизор включен!
Запущена печать на принтере...
Телевизор выключен..

І на завершення треба сказати, що інтерфейси в цій якості особливо широко використовуються в різних графічних API - AWT, Swing, JavaFX, де обробка подій об'єктів – елементів графічного інтерфейсу особливо актуальна.
