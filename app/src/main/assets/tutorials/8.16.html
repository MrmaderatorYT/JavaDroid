Змінні volatile

Останнє оновлення: 22.10.2025

Ключевое слово volatile предлагает механизм синхронизации доступа к полю экземпляра без блокировки. Зачем оно потрібно? При выполнении потіки можуть кэшировать значення змінних, а компилятор и процесор — переупорядочивать инструкции для оптимизации. Ключевое слово volatile є одним из инструментов для решения цих проблем. При оголошенийии поля с ключевым словом volatile, компилятор и виртуальная машина учитывают, что поле может одновременно обновляться другим потіком. При компиляции компилятор вставит соответствующий код, чтобы гарантировать, что зміна volatile-змінної в одном потіке буде видно из любого другого потіка, який считывает эту змінну.

Основная проблема, яку призвано решить слово volatile, - це проблема видимости. В соответствии с моделью памяти Java (Java Memory Model або JMM), каждый потік имїїт свою собственную "рабочую память" (work memory), которая часто реалізуєся як кэш процесора. Когда потік змінює значення змінної, он сначала может змінити его в своем локальном кэше. Другие потіки можуть не увидеть це зміна немедленно, оскільки они можуть читать "устаревшїї" значення из основной памяти або из "своих" кэшей.

Рассмотрим простой пример:


public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        Worker worker = new Worker();
        
        // Створюємо "Потік А" и запускуємо его. Он вызовет worker.run()
        Thread workerThread = new Thread(worker);
        workerThread.start(); // <-- Здесь запускуєся run()

        // Даем "Потіку А" немного времени, чтобы он вошел в цикл while(true)
        Thread.sleep(1000); // Ждем 1 секунду

        // "Потік Б" (главный потік) вызывает stop()
        worker.stop(); // <-- Здесь вызывается stop()

        // Ждем завершення "Потіка А"
        // Если флаг running не був volatile, join() может ждать вечно!
        workerThread.join(2000); // Ждем до 2 секунд

        if (workerThread.isAlive()) {
            System.out.println("--- РЕЗУЛЬТАТ: Потік А все ще работает! ---");
            System.out.println("Потік А не увидел зміна running = false.");
            System.exit(1); // Завершуємо програму, оскільки потік "завис"
        } else {
            System.out.println("--- РЕЗУЛЬТАТ: Потік А  успешно завершился. ---");
        }
    }
}

class Worker implements Runnable {

    // Флаг НЕ volatile
    private boolean running = true;

    @Override
    public void run() {
        
        // Цет код виконується в "Потіке А"
        System.out.println("Потік А: Начинаю работу...");
        while (running) {
            // Здесь может быть некоторая работа
            
            // JIT-компилятор с высокой вероятностью "закэширует" змінну running
            // в регистре процесора, оскільки она не меняется всередині цикла.
        }
        // Эта рядок может никогда не выполниться
        System.out.println("Потік А: Работа остановлена.");
    }

    public void stop() {
        // Цет код виконується в "Потіке Б" (главном потіке)
        System.out.println("Потік Б: Надсилаю команду на зупинку...");
        this.running = false;
    }
}

Здесь визначений клас Worker, який представляє реализацию інтерфейса Runnable - завданьу, которая может выполняться в потіках. В цем класе визначенийо два методу. Метод run() - це як раз реалізація Runnable. Цет метод буде выполняться в дочернем потіке (потік А), поки змінна running равна true. А второй метод - stop() призначений для сброса змінної running в false в главном потіке (потік Б).

В методі main() створюємо об'єкт Worker и запускуємо его виконання в дочернем потіке workerThread (условно "Потік А"):


Worker worker = new Worker();
        
// Створюємо "Потік А" и запускуємо его. Он вызовет worker.run()
Thread workerThread = new Thread(worker);
workerThread.start(); // <-- Здесь запускуєся run()

ПРи выполнении потіка workerThread фактически виконується метод run() класа Worker. Так як змінна running равна true, то потік workerThread (Потік А) бесконечно виконує цикл while

Далїї після небольшой задержки в 1 секунду сбрасываем у об'єкта Worker флаг running в false (чтобы Потік А завершил виконання) и за допомогою вызова методу join() ждем завершення потіка workerThread (Потіка А):


Thread.sleep(1000); // Ждем 1 секунду

// "Потік Б" (главный потік) вызывает stop()
worker.stop();

// Ждем завершення "Потіка А"
// Если флаг running не був volatile, join() может ждать вечно!
workerThread.join(2000); // Ждем до 2 секунд

Після цього потік workerThread по идїї повинен завершиться, и мы проверяем его статус за допомогою методу isAlive() и вивідим соответствующие сообщения на консоль:


if (workerThread.isAlive()) {
    System.out.println("--- РЕЗУЛЬТАТ: Потік А все ще работает! ---");
    System.out.println("Потік А не увидел зміна running = false.");
    System.exit(1); // Завершуємо програму, оскільки потік "завис"
} else {
    System.out.println("--- РЕЗУЛЬТАТ: Потік А  успешно завершился. ---");
}

Но якщо мы запустимо програму, то мы можемо отримати следующий консольный вивід:


Потік А: Начинаю работу...
Потік Б: Надсилаю команду на зупинку...
--- РЕЗУЛЬТАТ: Потік А все ще работает! ---
Потік А не увидел зміна running = false.

В чем же проблема? Потік А, скорїї всего, оптимизировал свой цикл. Он мог "решить", что раз змінна running не меняется *всередині* цикла, нет смысла каждый раз проверять її в основной памяти. Цет потік просто використовує свое кэшированное значення (true) и бесконечно долбится в бесконечном цикле.

Как це исправить? Изменим в класе Worker одну рядок:

private boolean running = true;

на

private volatile boolean running = true;

С volatile Потік А буде вынужден при каждой итерации цикла while проверять значення running в основной памяти. Как лише Потік Б запишет туда false, Потік А немедленно це увидит и выйдет из цикла.

Что робить volatile?

Ключевое слово volatile надає две ключевые гарантии:

  • Гарантия Видимости (Visibility)

    volatile — це, по сути, инструкция для JVM: "Никогда не кэшировать эту змінну на уровне потіка". И когда змінна оголошується як volatile, происходят следующие моменты:

    • Запис в volatile-змінну всегда производится немедленно в основную память.

    • Читання volatile-змінної всегда происходит напрямую из основной памяти, аннулируя любое кэшированное значення.

    Так, якщо в примере выше змінна running бува бы оголошенийа як volatile (volatile boolean running = true), Потік А при каждой проверке while (running) був бы вынужден обращаться к основной памяти. Как лише Потік Б записал бы false в основную память, Потік А немедленно бы це увидел и завершил цикл.

  • Гарантия Упорядочивания (Happens-Before)

    volatile предотвращает визначенийные типи переупорядочивания инструкций, которые можуть производить компиляторы и процесоры для повышения производительности. volatile же встановлює отношение "happens-before":

    • Запис в volatile-змінну происходит після всех предыдущих чтений и записей в коді.

    • Читання volatile-змінної происходит до (happens-before) всех післядующих чтений и записей.

    Це означает, что volatile действует як "барьер памяти". Все, что буво до записи в volatile-змінну, гарантированно буде видно любому потіку, який після записи обратился к volatile-змінної.

Атомарность и volatile

Варто відзначити, что volatile не гарантирует атомарность операцій. Підзьмем класический пример с счетчиком:


private volatile int counter = 0;

public void increment() {
    counter++; // ОПАСНО!
}

Казалось бы, volatile повинен помочь, но операція counter++ — це не один шаг. Це три шага:

  1. Читання: читаємо текущїї значення counter (из основной памяти, т.к. volatile).

  2. Зміна: увеличиваем це значення (наприклад, с 0 до 1).

  3. Запис: записуємо новое значення counter (в основную память, т.к. volatile)

Предположим ситуацию, что два потіка (А и Б) вызывают increment() одновременно:

  1. Потік А читає counter (значення 0).

  2. Потік Б читає counter (значення 0).

  3. Потік А змінює свое значення на 1 и записує 1 в counter.

  4. Потік Б змінює "свое" значення (которое все ще 0) на 1 и записує 1 в counter.

Таким чином, мы можемо отримати некорректный результат, а volatile соответственно не смог защитить от "состояния гонки" (race condition) во час составной операції.

Когда використовувати volatile

volatile можна назвать "облегченным" механизмом синхронизации, який застосовується, когда треба обеспечить видимость изменений змінних між потіками. Однак его йде використовувати лише тогда, когда виконуються оба следующих условия:

  • Изменения змінної не зависят от її текущего значення (т.е. вы не робитье counter++ або x = x + 1).

  • Запис в змінну производит лише один потік, а її значення считывают один або декілька других потіков.
