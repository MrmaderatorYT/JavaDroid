Змінні volatile

Останнє оновлення: 22.10.2025

Ключевое слово volatile предлагает механизм синхронизации доступа к полю экземпляра без блокировки. Зачем оно потрібно? При выполнении потоки можуть кэшировать значения переменных, а компилятор и процессор — переупорядочивать инструкции для оптимизации. Ключевое слово volatile є одним из инструментов для решения цих проблем. При оголошенийии поля с ключевым словом volatile, компилятор и виртуальная машина учитывают, что поле может одновременно обновляться другим потоком. При компиляции компилятор вставит соответствующий код, чтобы гарантировать, что изменение volatile-змінної в одном потоке буде видно из любого другого потока, який считывает эту переменную.

Основная проблема, яку призвано решить слово volatile, - це проблема видимости. В соответствии с моделью памяти Java (Java Memory Model або JMM), каждый поток имїїт свою собственную "рабочую память" (work memory), которая часто реализуется як кэш процессора. Когда поток змінює значение змінної, он сначала может изменить его в своем локальном кэше. Другие потоки можуть не увидеть це изменение немедленно, оскільки они можуть читать "устаревшїї" значение из основной памяти або из "своих" кэшей.

Рассмотрим простой пример:


public class Program {
 
    public static void main(String[] args) throws InterruptedException{
         
        Worker worker = new Worker();
        
        // Создаем "Поток А" и запускаем его. Он вызовет worker.run()
        Thread workerThread = new Thread(worker);
        workerThread.start(); // <-- Здесь запускаєся run()

        // Даем "Потоку А" немного времени, чтобы он вошел в цикл while(true)
        Thread.sleep(1000); // Ждем 1 секунду

        // "Поток Б" (главный поток) вызывает stop()
        worker.stop(); // <-- Здесь вызывается stop()

        // Ждем завершения "Потока А"
        // Если флаг running не був volatile, join() может ждать вечно!
        workerThread.join(2000); // Ждем до 2 секунд

        if (workerThread.isAlive()) {
            System.out.println("--- РЕЗУЛЬТАТ: Поток А все ще работает! ---");
            System.out.println("Поток А не увидел изменение running = false.");
            System.exit(1); // Завершаем программу, оскільки поток "завис"
        } else {
            System.out.println("--- РЕЗУЛЬТАТ: Поток А  успешно завершился. ---");
        }
    }
}

class Worker implements Runnable {

    // Флаг НЕ volatile
    private boolean running = true;

    @Override
    public void run() {
        
        // Цет код виконується в "Потоке А"
        System.out.println("Поток А: Начинаю работу...");
        while (running) {
            // Здесь может быть некоторая работа
            
            // JIT-компилятор с высокой вероятностью "закэширует" переменную running
            // в регистре процессора, оскільки она не меняется внутри цикла.
        }
        // Эта строка может никогда не выполниться
        System.out.println("Поток А: Работа остановлена.");
    }

    public void stop() {
        // Цет код виконується в "Потоке Б" (главном потоке)
        System.out.println("Поток Б: Отправляю команду на остановку...");
        this.running = false;
    }
}

Здесь определен класс Worker, який представляє реализацию інтерфейса Runnable - задачу, которая может выполняться в потоках. В цем классе определено два метода. Метод run() - це як раз реализация Runnable. Цет метод буде выполняться в дочернем потоке (поток А), поки переменная running равна true. А второй метод - stop() призначений для сброса змінної running в false в главном потоке (поток Б).

В методе main() создаем об'єкт Worker и запускаем его выполнение в дочернем потоке workerThread (условно "Поток А"):


Worker worker = new Worker();
        
// Создаем "Поток А" и запускаем его. Он вызовет worker.run()
Thread workerThread = new Thread(worker);
workerThread.start(); // <-- Здесь запускаєся run()

ПРи выполнении потока workerThread фактически виконується метод run() класса Worker. Так як переменная running равна true, то поток workerThread (Поток А) бесконечно выполняет цикл while

Далїї после небольшой задержки в 1 секунду сбрасываем у об'єкта Worker флаг running в false (чтобы Поток А завершил выполнение) и за допомогою вызова метода join() ждем завершения потока workerThread (Потока А):


Thread.sleep(1000); // Ждем 1 секунду

// "Поток Б" (главный поток) вызывает stop()
worker.stop();

// Ждем завершения "Потока А"
// Если флаг running не був volatile, join() может ждать вечно!
workerThread.join(2000); // Ждем до 2 секунд

После цього поток workerThread по идїї повинен завершиться, и мы проверяем его статус за допомогою метода isAlive() и выводим соответствующие сообщения на консоль:


if (workerThread.isAlive()) {
    System.out.println("--- РЕЗУЛЬТАТ: Поток А все ще работает! ---");
    System.out.println("Поток А не увидел изменение running = false.");
    System.exit(1); // Завершаем программу, оскільки поток "завис"
} else {
    System.out.println("--- РЕЗУЛЬТАТ: Поток А  успешно завершился. ---");
}

Но якщо мы запустим программу, то мы можемо отримати следующий консольный вывод:


Поток А: Начинаю работу...
Поток Б: Отправляю команду на остановку...
--- РЕЗУЛЬТАТ: Поток А все ще работает! ---
Поток А не увидел изменение running = false.

В чем же проблема? Поток А, скорїї всего, оптимизировал свой цикл. Он мог "решить", что раз переменная running не меняется *внутри* цикла, нет смысла каждый раз проверять її в основной памяти. Цет поток просто использует свое кэшированное значение (true) и бесконечно долбится в бесконечном цикле.

Как це исправить? Изменим в классе Worker одну строку:

private boolean running = true;

на

private volatile boolean running = true;

С volatile Поток А буде вынужден при каждой итерации цикла while проверять значение running в основной памяти. Как лише Поток Б запишет туда false, Поток А немедленно це увидит и выйдет из цикла.

Что робить volatile?

Ключевое слово volatile надає две ключевые гарантии:

  • Гарантия Видимости (Visibility)

    volatile — це, по сути, инструкция для JVM: "Никогда не кэшировать эту переменную на уровне потока". И когда переменная оголошується як volatile, происходят следующие моменты:

    • Запись в volatile-переменную всегда производится немедленно в основную память.

    • Чтение volatile-змінної всегда происходит напрямую из основной памяти, аннулируя любое кэшированное значение.

    Так, якщо в примере выше переменная running бува бы оголошенийа як volatile (volatile boolean running = true), Поток А при каждой проверке while (running) був бы вынужден обращаться к основной памяти. Как лише Поток Б записал бы false в основную память, Поток А немедленно бы це увидел и завершил цикл.

  • Гарантия Упорядочивания (Happens-Before)

    volatile предотвращает определенные типы переупорядочивания инструкций, которые можуть производить компиляторы и процессоры для повышения производительности. volatile же устанавливает отношение "happens-before":

    • Запись в volatile-переменную происходит после всех предыдущих чтений и записей в коде.

    • Чтение volatile-змінної происходит до (happens-before) всех последующих чтений и записей.

    Це означает, что volatile действует як "барьер памяти". Все, что буво до записи в volatile-переменную, гарантированно буде видно любому потоку, який после записи обратился к volatile-змінної.

Атомарность и volatile

Варто відзначити, что volatile не гарантирует атомарность операций. Підзьмем классический пример с счетчиком:


private volatile int counter = 0;

public void increment() {
    counter++; // ОПАСНО!
}

Казалось бы, volatile повинен помочь, но операция counter++ — це не один шаг. Це три шага:

  1. Чтение: читаем текущїї значення counter (из основной памяти, т.к. volatile).

  2. Изменение: увеличиваем це значение (наприклад, с 0 до 1).

  3. Запись: записываем новое значение counter (в основную память, т.к. volatile)

Предположим ситуацию, что два потока (А и Б) вызывают increment() одновременно:

  1. Поток А читає counter (значение 0).

  2. Поток Б читає counter (значение 0).

  3. Поток А змінює свое значение на 1 и записує 1 в counter.

  4. Поток Б змінює "свое" значение (которое все ще 0) на 1 и записує 1 в counter.

Таким чином, мы можемо отримати некорректный результат, а volatile соответственно не смог защитить от "состояния гонки" (race condition) во час составной операции.

Когда використовувати volatile

volatile можна назвать "облегченным" механизмом синхронизации, який застосовується, когда треба обеспечить видимость изменений переменных между потоками. Однак его йде використовувати лише тогда, когда выполняются оба следующих условия:

  • Изменения змінної не зависят от її текущего значения (т.е. вы не робитье counter++ або x = x + 1).

  • Запись в переменную производит лише один поток, а її значення считывают один або декілька других потоков.
