Заборона наслідування та перевизначення методів

Останнє оновлення: 24.09.2025

Хотя наследование очень интересный и эффективный механизм, но в некоторых ситуациях его применение может быть нежелательным. И в цем случае можна запретить наследование за допомогою ключевого слова final. Например:


final class Person { }

Якби клас Person був би визначений таким чином, то наступний код був би помилковим і не спрацював, тому що ми тим самим заборонили успадкування:


class Employee extends Person{ }  // ! Ошибка - унаследовать от класса Person нельзя

Кроме запрета наследования можна також запретить переопределение отдельных методів. Наприклад, в примере выше переопределен метод print(), запретим его переопределение:


class Person {
     
    private String name;

    Person(String name){ this.name = name;  }

    // запрет переопределения
    final void print(){
         
        System.out.println("Name: " + name);
    }
}

І тут клас-спадкоємець зможе перевизначити метод print:


class Employee extends Person{

    private String company;

    Employee(String name, String company){
        super(name); 
        this.company = company;
    }
    // ! Ошибка - переопределить метод нельзя
    void print(){
        super.print();
        System.out.println("Company: " + company);
    }
}

Запрет наследования класса або переопределения метода за допомогою final может быть полезным, якщо мы хочемо гарантировать, что семантика класса або метода не буде изменена в производном классе, то есть класс або метод буде действовать ровно так, як он був изначально определен. Поскольку якщо внутри базового класса вызываются его методы, то переопределение цих методів в производном классе может нарушить стандартную и ожидаемую работу цього класса. Рассмотрим небольшой пример:


public class Program{ 
       
    public static void main (String args[]){

        Employee bob = new Employee("Bob", "Google");
    }
}
class Person {
     
    private String name;

    Person(String name){ 

        this.name = name;  
        print();
    }

    void print(){
         
        System.out.println("Name: " + name);
    }
}

class Employee extends Person{

    private String company;

    Employee(String name, String company){

        super(name); 
        this.company = company;
    }
    @Override
    void print(){
        super.print();
        System.out.println("Company: " + company);
    }
}

В конструкторе класса Person вызывается метод print() для вывода информации о созданном об'єкте. Но в классе Employee переопределяем цей метод, чтобы також вывести значение поля company.

Оскільки у конструкторі Employee ми вже викликаємо конструктор базового класу Person, нам не потрібно вдруге викликати метод print:


Employee(String name, String company){

    super(name);   // здесь вызывается метод print
    this.company = company;
}

Однак подивимося на консольне виведення програми:


Name: Bob
Company: null

Конструктор базового класса Person, где вызывается метод print, отработал ДО установки поля company. В итоге при выводе на консоль виводиться некорректная информация.

Вызов метода базового класса в конструкторе - це одна из распространенных причин запрета переопределения метода. И в цем случае метод print в классе Person мы можемо определить як final:


public class Program{ 
       
    public static void main (String args[]){

        Employee bob = new Employee("Bob", "Google");
    }
}
class Person {
     
    private String name;

    Person(String name){ 
        this.name = name;  
        printPerson();
    }
    // запрет переопределения
    final void printPerson(){
         
        System.out.println("Name: " + name);
    }
}

class Employee extends Person{

    private String company;

    Employee(String name, String company){

        super(name);  // в цей точке в базовом классе вызывается метод printPerson 

        this.company = company;
        printEmployee();
    }

    void printEmployee(){
        System.out.println("Company: " + company);
    }
}

І тут отримаємо коректний висновок:


Name: Bob
Company: Google

Варто відзначити, що проблема була пов'язана насамперед з тим, що виклик конструктора базового класу виконувався до решти коду (установки поля company). І в до версії Java 25 ми могли викликати конструктор базового класу тільки на початку конструктоора похідного класу до решти коду:


Employee(String name, String company){

    super(name); 
    this.company = company;
}

Однак, починаючи з Java 25, можна викликати конструктор базового класу в будь-якому місці конструктора похідного класу:


Employee(String name, String company){

    this.company = company;
    super(name); 
}

Конкретно эта деталь в данном случае бы решила проблему, тем не менїї в болїї сложных программах нам треба учитывать все возможные зависимости, которые можуть повлять на створення об'єкта базового класса. Тому в конструкторе лучше вызывать лише методы с модификаторами final або private.
